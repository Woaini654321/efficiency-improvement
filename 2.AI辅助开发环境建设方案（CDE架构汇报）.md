# AI 辅助开发环境建设方案

> 主题：用 AI 大模型辅助开发进行研发提效——开发环境基础设施的目标架构与迁移路径
> 整理：Leon Yan 晏玉龙 · 2026-07-20

---

## 一、背景与目标

公司计划引入 AI 大模型（Claude Code 等）辅助日常开发。基础设施要同时满足四点：

1. **环境统一**：编译器、工具链、AI 助手版本一致，新人零配置入场；
2. **知识共享**：项目记忆、编码规范、企业框架经验可沉淀、可复用；
3. **安全可控**：代码、凭据不外泄，AI 的操作可审计、可限权；
4. **并发无冲突**：多人多任务同时进行，互不干扰。

## 二、现状方案与问题分析

**现状**：远程服务器部署 Docker，容器内安装编译器、Claude Code、git 等工具；开发者通过网页和脚本与容器通信，控制容器内工具；多人共享容器与项目记忆。

**方向判断**：「远程统一环境」这个大方向是对的——业界同类形态叫 **CDE（Cloud Development Environment，云开发环境）**，已有成熟产品。问题不在"远程 Docker"，而在**"多人共享同一个容器/同一份工作区"**这一个设计点上，它是下表所有问题的共同根因：

| # | 问题 | 根因分析 | 后果 |
|---|------|----------|------|
| 1 | 安全风险大 | 所有人的凭据（git token、API key）与代码在同一容器内，彼此可见 | 任何一人的误操作或 **prompt 注入**，即可读取/外带全部人的代码与凭据；爆炸半径 = 全team |
| 2 | 同时读写同一份文件 | 多人/多 AI 会话共用同一工作区，用文件系统承担了本该由 git 承担的并发汇聚职责 | 改动互相覆盖、构建互相污染、AI 上下文互相串味 |
| 3 | 共享记忆易脏写 | 记忆放共享可写卷（如共享 `~/.claude`） | 并发写坏记忆文件；无版本、无评审、无回滚 |
| 4 | 自研控制面维护成本 | 网页+脚本控制容器属自研 CDE 控制面 | 重复造轮子，权限/审计/会话管理都要自己补 |
| 5 | 操作不可审计、权限一刀切 | 共享容器内无法区分"谁让 AI 干了什么" | 出事无法定责，也无法按角色收紧 AI 权限 |

## 三、目标架构：共享镜像与规范，隔离工作区，git 汇聚

成熟方案没有一家让多人读写同一份文件。标准形态一句话：**共享的是"黄金镜像 + 规范/记忆（走 git）"，隔离的是"每人/每任务的工作区"，产出靠 git MR 汇聚串行化。**

```
┌────────────────────────────────────────────────────────┐
│  黄金镜像（统一维护，即现有 Docker 镜像的升级）           │
│  编译器/JDK/Node + Claude Code + git + 代理白名单配置    │
│  + 预装公司 skill 插件市场（vue-coding / java-coding…） │
└───────────────┬────────────────────────────────────────┘
                │ CDE 平台按需实例化（Coder / code-server）
     ┌──────────┼──────────┐
     ▼          ▼          ▼
  开发者A容器  开发者B容器  CI任务容器      ← 每人/每任务独立，
  (own clone)  (own clone) (headless agent)   互不可见、随用随建
     │          │          │
     └────── MR / PR ──────┘
                ▼
┌────────────────────────────────────────────────────────┐
│  GitLab（git.quectel.com）                              │
│  受保护分支 + 评审 + CI 门禁 = 并发写的唯一串行化点       │
│  仓库内 CLAUDE.md / MEMORY.md / skill = 版本化的共享记忆 │
└────────────────────────────────────────────────────────┘
```

### 关键设计点

**1）并发正解 = git，不是文件锁**
每个会话在自己的 clone（或同容器内 git worktree）上工作，产出以 MR 提交；冲突由 merge 机制解决——三十年验证过的并发模型。现状的"同时读写同一份文件"，本质是让文件系统干了 git 的活。

**2）安全分层（改成每人一容器后叠加）**

| 层 | 措施 |
|---|------|
| 凭据 | 不进镜像：git 用 GitLab OAuth/OIDC 下发的**短时效 token**；LLM API key 由网关统一注入并计量审计 |
| 网络 | 容器出网走 egress 代理**白名单**（仅 Nexus、GitLab、LLM 网关），防数据外带 |
| 容器 | rootless 运行、不挂 docker.sock；需更强隔离时用 gVisor / Kata |
| AI 权限 | Claude Code permission mode 按角色下发（受管 settings 打进镜像）；操作日志集中审计 |
| 定责 | 每容器绑定唯一身份（GitLab OAuth），"谁让 AI 干了什么"天然可查 |

**3）共享记忆正解 = git 版本化，不是共享卷**

| 层级 | 载体 | 说明 |
|---|------|------|
| 项目级 | 仓库内 `CLAUDE.md` / `MEMORY.md` / `AGENTS.md` | clone 即得、带评审带历史带回滚（sales-lead-hub 项目已实战验证此模式） |
| 公司级 | 内网 skill/插件市场（quectel-plugin-marketplace） | 企业框架经验做成 skill 预装进黄金镜像，全员复用 |
| 个人级 | 各自容器的私有卷 | 不共享，避免脏写 |

## 四、方案选型对比

| 方案 | 形态 | 内网适配 | 建议 |
|---|------|:---:|------|
| **Coder（自托管）** | 开源 CDE：模板化镜像，每人一键开独立工作区，浏览器 IDE + SSH，**内置 AI Agent/Tasks 可直接编排 Claude Code** | 高 | **推荐主选**，直接替换自研网页控制面 |
| code-server / JetBrains Gateway | 每人一容器跑浏览器 IDE，Claude Code 装容器内 | 高 | **推荐起步**，最小改造，先跑通模式 |
| Gitpod 自托管（Ona）/ Eclipse Che | 同类 CDE，按仓库快照秒开 | 中 | 部署较重，备选 |
| GitLab CI + headless agent | `claude -p` 无头模式按任务起容器，产出 MR | 高 | **与 CDE 互补**，用于批量任务（批量迁移/修 lint/巡检） |
| Claude Code on the Web | Anthropic 托管沙箱，每任务独立 VM | 低 | 代码需出内网，暂不适用 |

## 五、迁移路线（三阶段，现有投入不推翻）

| 阶段 | 动作 | 复用现状 | 退役内容 |
|---|------|----------|----------|
| **P1 止血**（1~2 周） | ① 现有镜像升级为黄金镜像（加代理白名单、受管 settings、预装 skill 市场）；② 改"共享一容器"为**每人一容器**（先用 docker compose/脚本 + code-server 过渡）；③ 记忆迁进仓库（CLAUDE.md/MEMORY.md 模板化推广） | 服务器、镜像、脚本 | 共享工作区、共享记忆卷 |
| **P2 平台化**（1~2 月） | ① 部署 Coder，对接 GitLab OAuth；② 凭据改短时效 token + LLM 网关统一注入；③ 建立 MR 汇聚 + 受保护分支 + CI 门禁流程；④ 操作审计落库 | 黄金镜像直接变 Coder 模板 | 自研网页控制面 |
| **P3 规模化**（持续） | ① GitLab CI headless agent 承接批量任务；② skill 市场持续沉淀企业框架经验（每个项目复盘反哺）；③ 按 token 用量/MR 通过率做提效度量 | 全部 | — |

## 六、风险与对策

| 风险 | 对策 |
|---|------|
| 每人一容器资源开销上升 | CDE 平台支持闲置自动休眠/回收；工作区秒级重建（一切状态在 git） |
| 开发者习惯迁移成本 | P1 阶段浏览器 IDE 体验与现状网页接近；SSH 通道保留给重度用户 |
| LLM 网关单点 | 网关做高可用；key 轮换与限流在网关层统一处理 |
| 记忆入库后含敏感信息 | CLAUDE.md 走 MR 评审；CI 加密钥扫描门禁（明文口令禁入库，sales-lead-hub 已验证此规范） |

## 七、一句话总结

> 保留"远程统一环境"的方向和已有镜像投入，把唯一的病灶——**共享工作区**——改为"黄金镜像 → 每人/每任务隔离容器 → git MR 汇聚"，记忆走仓库版本化、经验走 skill 市场，控制面交给成熟 CDE 平台（Coder），安全与并发两大问题即从根上消解。
