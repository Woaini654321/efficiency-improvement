# 用 Claude Code 做竖切联调与 Mock 批量切真（从"库能建"到"数据是真的"）· 方法论复盘

> 一次真实会话链的复盘：**在"数据库从 0→1 建好、双角色评审收口"之后，把项目推进到"真实数据在前后端之间流动"——先用 TDD 把首个模块（requirement）做成完整竖切并用真实 SSO 账号打穿；途中钉死三类只有联调才暴露的黑盒陷阱（数据源命名空间、UAA 角色不可用、序列化契约三坑）；然后把竖切固化成"配方"，写出全模块 Mock→真实的迁移方案；先做一次横切地基（模块级 mock 开关/批修 total/分页基类），再按同构度批量复制配方（opportunity→category→auditLog…）；每个模块都以"浏览器实测 + JDBC 核库"收口，并在实测中抓住单测全绿但库是错的真 bug。**
> 载体仍是「销售商机互助平台」（Quectel-code 企业框架 + QMonoX 前端 + Spring Boot 2.7 + MyBatis-Plus + UAA SSO）。本文是 0→1 系列的**第七篇**，承接《用 Claude Code 做契约收口与数据库从 0→1 方法论》——**上一篇讲"把库稳稳建起来"，这一篇讲"让真数据流过每一层，并把 20 个模块的 mock 有序退场"**，让 0→1 链路走到"可验证的活系统"。

## 一、结论先行（TL;DR）

**"页面能跑 + 库能建"不等于"系统是活的"。中间隔着两段最容易翻车的路：首个模块的真实联调（一堆只有跑起来才暴露的黑盒陷阱），和其余十几个模块的批量切真（一堆重复劳动里藏着的结构性坑）。**

- 🔩 **首个模块做成"竖切配方"，其余模块只写 delta**：requirement 竖切验证过的 12 类文件（DAO/DTO/VO/Convert/ErrorCode/Service/Controller/三层测试/前端切换）固化成清单；opportunity 照配方复制，任务描述里**只写它独有的 delta**（状态机/生成列/代发布），不重复 TDD 步骤。
- 🧱 **批量复制前，先做一次横切地基**：全局单布尔 mock 开关改成**模块级白名单**（否则每接一个模块都要重演"绕开开关"）；15 个 adapter 的 `total` 类型批修一次；分页入参抽基类。**每个模块都会踩的坑，做一次 vs 做 N 次。**
- 📜 **契约以前端"实发 payload"为准，不是类型声明、更不是 PRD**：读 `buildParams()` 才发现存草稿/发布共用一套 payload 且**没有 status 字段**（后端无从区分）、`industry/keywords` 是页面不回显的 write-only 残留、列表页实发 `pageSize:999` 会撞后端 `@Max(500)`。**类型文件会撒谎，页面代码不会。**
- 🔬 **联调是某类 bug 唯一的暴露层，且改完必须核库**：MP `updateById` 对 null 字段静默跳过——"恢复上架清空 archived_by"单测断言实体全绿，**库里却残留旧值**；只有"浏览器操作 + JDBC 直查"这一层能抓到。接口返回 200 不算数，**落库的行才算数**。
- 🚪 **Mock 要设计"退场机制"，不是一键关掉**：按模块从白名单摘除（后端就绪一个摘一个）；mock 文件保留可回退；**页面级 mock 碎片**（硬编码分类树/选人列表）防腐层管不到，要单独清点——它们的假 id 会被后端 `parseIds` 直接拒。

> 一句话方法论：**首个模块打穿并固化成配方；横切坑先做一次地基；契约向页面实发代码取证；每个模块以"真实账号操作 + 数据库核行"收口；mock 按模块有序退场、页面级碎片单独清剿；只有真实联调能抓的 bug，就别指望测试金字塔的下层。**

## 二、正确的主线流程（可照抄）

```
竖切线（首个模块 requirement 打穿）
  1. TDD 逐层实施：Convert 契约测试(离线断言 snake_case/日期格式) → Service 单测 → 真库集成测试
  2. 集成测试防假绿：@Tag("integration") + pom 属性化 excludedGroups；类名必须以 Test 结尾（*IT 不被 surefire 扫描，报 BUILD SUCCESS 但 Tests run: 0）
  3. 联调陷阱逐个钉死：
     - 数据源命名空间：框架只读 quectel.mysql.{server,port,database,uname,pwd}，写 spring.datasource.* 完全不读
     - 401 证明不了路由存在：TokenValidationFilter 在 Spring MVC 路由之前拦截——判断端点要用带 token 的请求
     - UAA 无业务角色：/me 实测返回 100+ 不透明角色 ID → @PreAuthorize 永不放行 → 鉴权改查本地 sys_user.role（fail-closed）
     - 权限异常抛 BaseException(FORBIDDEN)，禁 AccessDeniedException（框架兜成 500）
  4. Playwright 真实 SSO 登录全流程 + JDBC 核库（create_by=真实 UAA id、快照字段、关联行）
  5. 踩过的坑立即固化 memory（序列化三坑、UAA 角色、乐观锁需自补）

方案线（全模块切真的作战图）
  6. 先盘点再写方案：数一遍接口总数/已完成数；识破两个误判——
     - "写接口还没切"是假象：mockRequest 只包了读接口，写接口从来就是裸奔（点下去就 404）→ 没有平滑迁移，只有补齐
     - "都有表"是假象：6 个模块 20 个接口在 schema 里一张表都没有 → 属产品决策，单独成 Phase 阻塞项
  7. 把竖切固化成 12 步配方；各模块任务只写 delta；按"同构度"排 Phase（最像的先做，验证配方可复制）
  8. 聚合统计类（home/dashboard/sla）放最后：上游表没有真数据时它们只能返回一堆 0，无法验证

地基线（Phase 0，横切一次性）
  9. mock 开关改模块级白名单 Set；调用点批量注入模块键；已接通模块回归走开关（不在名单=直连）
  10. 批修全部 adapter 的 total: Number(data.total ?? 0)（脚手架模板每个模块都有这行）
  11. 抽 BasePageDTO——字段与前端 PaginationParams 契约逐字对齐（恰好 4 个，keyword 是模块扩展，不进基类）

复制线（Phase 1+，每模块一循环）
  12. 读前端实发契约（types.ts + buildParams() + 调用点 grep）→ 发现契约缺口当场补（status 字段/pageSize 夹逼）
  13. 调用方取证做 YAGNI：grep 无任何业务页调用的端点（opportunity/delete、employee/page）→ 不做，记录
  14. 按配方产出 → 离线全绿 → 真库集成测试 → 前端切真+门禁 → 浏览器实测 → JDBC 核行
  15. 实测抓到的 bug：修复 + 真库集成测试钉死 + 固化 memory（MP null 更新静默跳过 → FieldStrategy.ALWAYS）
  16. 页面级 mock 碎片清剿：硬编码分类树先换真实 DB id（过渡态、注释标退场条件），依赖接口就绪后切远程加载删除常量
```

## 三、关键方法与技巧复盘（本文精华）

### 技巧 1 · 竖切配方化：第一个模块慢慢打穿，其余模块只写 delta
requirement 竖切花了整个会话（TDD + 联调 + 排障），但它的产出不只是 4 个接口，而是**一份验证过的配方**：12 类文件怎么写、测试怎么分层、前端怎么切换、每一步的验收命令。写全模块迁移方案时，把配方独立成章，每个模块的任务**只写"相对配方的 delta"**——opportunity 的 delta 是三态状态机/JSON 附件列/代发布；interaction 的 delta 是生成列只读映射/点赞幂等/软删语义。**delta 才是需要动脑的部分；配方保证不动脑的部分不出错。** 实际执行验证了这一点：opportunity 全套后端从开写到离线全绿，一次编译通过。

### 技巧 2 · 横切地基先行——"每个模块都会踩一次的坑"值得专门一个 Phase
盘点时识别出三件事是**所有模块共有**的：① 全局单布尔 mock 开关（requirement 当初被迫绕开它单独直连，不改则每个模块都要重演）；② 脚手架模板给每个 adapter 都生成了 `total: data.total ?? 0`（后端 Long→String 会把 total 变字符串，切真逐个中招）；③ 每个 PageDTO 都要重复分页四字段。**Phase 0 专门做这三件事，0 个接口产出，却让后面每个模块的切换成本从"一套动作"降为"从名单删一行"。** 判据：凡是"接第 N 个模块时还会再做一遍"的动作，都应该提前抽成地基。

### 技巧 3 · 契约向"实发 payload"取证——类型文件会撒谎，页面代码不会
后端 DTO 的字段清单，**唯一可信来源是页面提交函数实际构造的对象**（`buildParams()`），不是 types.ts 的声明、不是 PRD。本次靠读实发代码抓到四个契约缺口：**存草稿和发布共用 payload 且没有 status**（后端无从区分，当场给契约补 `status` 字段并在后端按状态条件校验——草稿只要标题、发布须分类 1~5+正文，与前端行为逐字对齐）；`industry/keywords` 是**write-only 残留**（出参无此字段、页面不回显、schema 无列→后端刻意不声明，Jackson 静默忽略，不落库）；列表页实发 `pageSize:999`（撞 `@Max(500)`，前端收敛为 500）；changeStatus 实发值是 `'archived'/'published'`（状态机取值直接从调用点抄）。上一篇的"契约收口"发生在 mock 阶段，这一篇证明：**切真前还要再向实发代码核对一次——mock 阶段的收口挡不住页面后来加的字段。**

### 技巧 4 · YAGNI 要靠调用方取证，不靠感觉
api 文件里导出的函数 ≠ 需要实现的接口。`opportunity/delete` 在全部页面里 grep 不到任何调用（死代码）；`employee/page` 唯一调用方是 ai-request-guard-demo 演示模块。两者都**判定不做**，各省一套后端竖切，且在 Controller javadoc 和方案里记录判定依据。**判据：grep 调用方，零业务页调用 → 不做；未来要用时再按配方补，成本可控。** 反过来也成立：changeStatus 有两个页面调用 → 必须做，且语义（谁下架谁恢复）从 schema 注释取证。

### 技巧 5 · 联调是某类 bug 唯一的暴露层——改完必须核库，200 不算数（本篇最硬核）
本次抓到的最典型 bug：恢复上架要把 `archived_by` 清回 NULL，service 写了 `d.setArchivedBy(null)`，**单测断言实体为 null 通过、接口返回 200、页面显示正常——但 JDBC 直查发现库里残留旧下架人**。根因是 MyBatis-Plus `updateById` 默认 NOT_NULL 策略**静默跳过 null 字段**。这类 bug 的结构特征是：**测试金字塔的每一层都绿，只有"真实操作 + 数据库核行"这一层是红的。** 修复三件套：该列 `@TableField(updateStrategy = FieldStrategy.ALWAYS)`；**真库集成测试钉死**（"若残留说明 ALWAYS 被移除"）；固化 memory（凡"把某列清回 NULL"语义的字段建 DO 时就标 ALWAYS）。方法论：**每个模块的验收动作里，"JDBC 核对落库行"与"门禁绿"同权重**——核 create_by 是不是真实 UAA id、快照字段是否来自本地档案、关联表行数是否正确、置 null 是否真的 NULL。

### 技巧 6 · 环境问题当刑侦做：进程被谁杀、请求被谁吞，都要查到凶手
联调期两类"灵异现象"都靠取证破案：① 后端服务每次启动后 3~5 分钟被外部终止（exit -1）——查 `Win32_Process` 的 CommandLine 找到一个残留的旧 start-server 僵尸实例在按"固定端口杀进程树"逻辑误杀新服务；根治 = 杀掉僵尸 + **改用 `Start-Process` 分离式启动**（脱离工具托管进程树，服务独立存活）。② `curl localhost` 返回 000——请求行出现绝对 URI，暴露 curl 走了公司代理，`--noproxy "*"` 解决（此前的就绪探测循环因此永远失败）。此外重申上一篇的教训并新增一条：**401 证明不了路由存在**（TokenValidationFilter 在 Spring MVC 之前拦截），判断"新端点是否已注册"必须用带 token 的请求或核对 target/classes 时间戳。**环境异常不绕过、不重试碰运气，查到进程/网络层的确切凶手再修。**

### 技巧 7 · Mock 的退场要设计成机制，且页面级碎片要单独清剿
"切真"不是把开关拨到 real 就完事，它是三层结构：① **接口层**——模块级白名单 Set，后端就绪一个摘一行，未就绪模块不受牵连（全局单布尔会让 19 个没后端的模块同时打向不存在的接口，404 风暴淹没正在联调模块的信号）；调用点保留 module 键，回退=加回名单。② **数据层**——mock JSON 保留不删，直到该模块联调稳定（回退成本≈0）。③ **页面层**——防腐层管不到页面内的硬编码常量：opportunity 表单的分类树是写死的 slug id（`'5g-module'`），切真后提交会被后端 `parseIds` 直接拒。处置分两步：先把硬编码值换成 **DB 真实 id 的过渡态**（注释标明"待 category/list 接通后删除本常量"），category 模块接通当天就兑现——两个发布表单的分类树改为远程加载，**树的 SSOT 回归 DB**，停用分类自动滤枝。用户中途一句"前端看到的还是 mock 数据"点破的正是第三层——**页面级碎片不清点，接口切得再干净用户也感知不到。**

### 技巧 8 · 数据感受悖论：越好看的页面越假，越空的页面越真——主动对齐认知
切真推进过半时，用户困惑"为什么页面都还是 mock"。诚实的拆解有三条：① 导航里最常打开的页面（首页/个人中心/dashboard/吐槽墙）恰好全是未切真模块——**菜单被未完成模块占据造成"什么都没变"的观感**；② 已切真页面只有联调写入的两三行真数据，而 mock 页面有几十条精心编造的记录——**数据越丰满越假、越空越真**，这个悖论必须主动说破；③ 快速分辨法交给用户：F12 网络面板，发 `/api/...` 请求的是真的。配套动作：提供「demo 种子数据」选项（把 mock JSON 转成真库行），让"真"和"好看"兼得。**进度的透明度也是交付物**——盘点表（已完成/剩余/为何看不出来）本身就该是给用户的正式产出。

## 四、给"竖切联调 + 批量切真"的协作姿势

| 姿势 | 说明 |
|---|---|
| **首模块配方化** | 第一个竖切慢慢打穿、固化 12 步配方；后续模块只写 delta。 |
| **地基先行** | 每个模块都会踩的坑（开关粒度/total 类型/分页基类）抽成 Phase 0 做一次。 |
| **契约向实发取证** | DTO 字段抄 buildParams()，不抄 types.ts/PRD；切真前再核一次实发。 |
| **YAGNI 靠 grep** | 端点做不做看调用方取证：零业务页调用→不做并记录依据。 |
| **验收=门禁+核库** | 接口 200 不算数；JDBC 核 create_by/快照/关联行/置 null 与门禁绿同权重。 |
| **实测 bug 三件套** | 修复 + 真库集成测试钉死 + 固化 memory，一个不缺。 |
| **环境刑侦** | 进程被杀查 CommandLine 找凶手；请求异常看请求行识破代理；不绕过不碰运气。 |
| **mock 分层退场** | 接口层白名单摘除、数据层保留可回退、页面层碎片单独清点（假 id 会被后端拒）。 |
| **聚合模块最后做** | home/dashboard/sla 依赖上游真数据，提前做只能验证一堆 0。 |
| **认知透明** | "为什么看起来还是 mock"值得一张正式盘点表；越好看越假要主动说破。 |

## 六、效率与成本的诚实备注

- **成本主要花在四处**：① requirement 竖切联调（含数据源/UAA 角色/序列化三坑的排障，占比最大但产出配方）；② 全模块方案盘点与编写（一次性）；③ opportunity 配方复制 + 浏览器实测（抓到 ALWAYS bug 的那轮实测最值）；④ 环境刑侦（僵尸进程/代理吞包，两次共约 20 分钟）。
- **省钱要点**：
  - **配方摊薄成本**：首模块最贵，但 opportunity/category 的边际成本断崖式下降——批量工作的正确投资姿势是"第一个做慢做透"。
  - **地基一次 vs 模块 N 次**：Phase 0 三件事没有任何接口产出，却是全程 ROI 最高的投入。
  - **门禁即验收**：每模块五关（离线/集成/前端门禁/实测/核库）当场过，不攒总账——bug 在当轮修比跨模块回溯便宜一个量级。
  - **实测坑升格记忆**：MP null 静默跳过、序列化三坑、UAA 角色——固化 memory 后，后续模块与后续会话直接命中，不复踩。
  - **知道何处停**：employee/delete 两处 YAGNI 各省一套竖切；无表 20 接口坚决不猜字段、留给产品拍板。
- **可推广性**：这套"首模块配方化→横切地基→实发契约取证→五关验收→mock 分层退场"的主线，适用于**任何"前端先行 + mock 数据的项目向真实后端批量迁移"**的场景——不限于本框架，可作团队 SOP。

## 七、一页纸总纲

> **首个模块用 TDD 打穿到真实 SSO + 真库，把过程固化成配方；每个模块都会重踩的坑抽成横切地基先做一次；后端契约向页面实发 payload 取证，类型声明和 PRD 都可能撒谎；端点做不做用调用方 grep 取证；每模块五关验收——离线测试、真库集成、前端门禁、浏览器实测、JDBC 核行，接口 200 不算数、落库的行才算数；实测抓到的 bug 用"修复+集成测试钉死+固化记忆"三件套处理；mock 按接口/数据/页面三层设计退场机制，页面级碎片单独清剿；聚合统计模块等上游有真数据再做；进度盘点表本身是交付物——越好看的页面越假、越空的页面越真，要主动说破。**
> 从"库能建"到"数据是真的"，真正的功夫也在两处：**把第一次联调的每个坑变成后面所有模块的免疫力；把"看起来成功"和"库里正确"严格区分开。** 系统是不是活的，不看页面多好看，看 MySQL 里那几行字段全对的数据。
