# Quectel 商机信息发布平台 · 跨岗位交叉评审报告

> 评审方式：产品经理 / 架构师 / 后端 / 前端 四个角色独立评审后交叉核对去重。
> 可信度标注：【四方】=四个角色均指出；【三方】/【两方】=多角色独立命中；【单方】=单一角色发现。
> 评审基准：living-docs/ 下 00~06 PRD、01 全局规约、feature-matrix、information-architecture、系统功能架构图、business-process、research、prototypes 全量原型。
> 日期：2026-07-18

## 0. 总体判断

文档在**页面级交互、字段字典、业务规则（66 条 BR / 32 条 EX / 15 页字段字典）**上颗粒度很细、质量较高。但存在三类**系统性风险**，会直接阻塞开工：

1. **范围处于两个未对齐的版本之间**：规格层（feature-matrix 42 FEAT / IA 9 实体 / 约 15 页详规）落后于原型层（系统功能架构图 30 页原型）。约 40% 的已画原型没有任何需求规格。
2. **多个"唯一真理源"并存且互相冲突**：权限矩阵、SLA 模型、可见性规则各有 2~3 处定义且内容不一致，SSOT 机制事实上未生效。
3. **红线能力缺基础支撑**：组织结构（部门/产品线/负责人）未建模、需求缺 deadline 字段、搜索引擎/消息队列选型与数据量级缺失，导致数据隔离、SLA 自动升级、搜索性能等承诺无法验证可实现性。

## 1. 高危问题（开工前必须澄清，P0 blocker）

### H-1 本期范围失控：六大功能群"在范围内但零需求定义" 【四方】
- **现象**：总纲 §2.3「本期做」列入讨论区/情报中心/工具助手/批量发布任务；系统功能架构图（30 页）进一步纳入 PC-18 讨论区系列、PC-20 情报系列、PC-15/16 公告、PC-23 批量任务；原型目录还多出 PC-19 吐槽墙、PC-21 会议任务、PC-22 我的任务。但 **feature-matrix 的 42 个 FEAT 无一覆盖这些功能**，6 个模块 PRD 也全部不涉及——无 FEAT-ID、无数据实体、无状态机、无字段、无业务规则、无验收标准。
- **典型**：PC-23（P0）PRD 正文仅一句「🚧 详细字段待原型细化后补充」；PC-22 全局导航不可达且零文档；PC-09（承载 P0 权限功能）反而有最完整 PRD 却无原型、无导航入口。
- **影响**：工作量可能较 42-FEAT 基线翻倍；这些功能与核心价值（商机发布 + 需求匹配）相关性弱，属典型"镀金"。范围/优先级严重失真，无法排期、无法建表、无法估容量。
- **对应决策**：D1

### H-2 SLA 体系多重冲突（最复杂的一处）【四方】
- **H-2a 计时模型二选一未定**：PRD/business-process 定义 SLA 按 urgency 自动计算首响时限（特急 2h/紧急 4h/普通 24h），PC-06 字段表**无**截止时间字段；但系统功能架构图、IA、PC-06 原型明确有"期望响应截止时间 DatePicker"，PC-05 显示"截止倒计时"。两者对"倒计时是 SLA 自动首响、还是用户自设 deadline"定义相反。
- **H-2b urgency 与 priority 双字段/双编码**：数据模型用 `urgency`(normal/urgent/critical)，SLA 监控页用 `priority`(P0/P1/P2)，ER 中只有 urgency 无 priority，二者映射从未写明。
- **H-2c SSOT 不完整**：01 手册 §5（唯一真理源）只定义"首响时限"，缺"解决时限"；解决时限(24h/48h/5 工作日)只在 business-process/IA/PC-13 出现——定义了却从不被监控使用。且引入"5 工作日"但无工作日历/节假日/时区/暂停恢复语义。
- **H-2d 升级链模型冲突**：01§5（特急三级/紧急二级/普通一级链，随优先级变级）vs 01§6.1 E-1（笼统三级 L1→L2→L3）vs IA PC-13（所有优先级统一 L0→L3，每级间隔=对应首响时限，固定三级）——SSOT 自身前后不一致。
- **H-2e 首响即停 vs Collecting 仍倒计时**：BR-056（收到首方案→responded 停止升级）与 BR-018（Pending 与 Collecting 均显示倒计时）矛盾——进入 Collecting 已代表首响，倒计时不应继续。
- **H-2f 升级链责任人无数据源**：L1 产品线负责人/L2 产品管理部/L3 管理层，但系统无"产品线→负责人"映射（见 H-3）。原型 PC-13 用 mock + 手动下拉选负责人，与"系统自动升级"矛盾。
- **对应决策**：D2、D3

### H-3 组织结构（部门/产品线/负责人）未建模，却被四项能力依赖 【两方：架构+后端】
- **现象**：ER 9 实体中无 Department、无 ProductLine 实体；`User.department_id`、`Opportunity.department_id` 是悬空外键；`invited_product_line_ids` 仅为 JSON 数组无实体；SLA 升级收件人无组织映射。
- **影响**：**数据隔离、SLA 自动升级、邀请产品线定向通知、可见范围**四项能力全部悬空，因为它们都依赖一个未定义的组织维度。
- **对应决策**：D3

### H-4 可见性规则自相矛盾 + 两套机制叠加未定义 【三方：后端+架构+PM】
- **H-4a 模块内矛盾**：BR-013「所有状态的需求对所有已登录用户可见」 vs FEAT-0209 可见范围（all/dept/personnel 限制）在同一 MOD-02 内直接互斥；AC-102 又要求"仅 A 部门可见"。
- **H-4b 两套机制叠加规则空白**：FEAT-0209 可见范围（每条需求级）与 FEAT-0502 部门数据隔离（全局管理员级）如何叠加（AND/OR、谁覆盖谁、默认策略）从未定义。
- **H-4c 商机/需求可见性不对称**：Opportunity 无 visibility_scope 字段（只有 department_id），需求有——两类内容可见性模型不一致。
- **安全维度**：跨部门默认不隔离，对含"客户场景"的销售需求（内部商业信息）是否可接受 500+ 人默认全可见？——正是 E-5 红线要防的数据泄漏。
- **对应决策**：D4

### H-5 权限矩阵三处定义且不一致，管理员权限存在死锁 【三方：架构+PM+前端】
- **H-5a 总数冲突**：总纲 §3.2（42 行，自称唯一真理源）vs IA 权限矩阵（38 行，缺 FEAT-0209~0212）vs 总纲 §6.2（"3 角色 × 42 功能"）vs IA/CHANGELOG（"38 功能"）。
- **H-5b 管理员权限无操作入口**：矩阵 FEAT-0206（关闭需求）/FEAT-0106（重新上架）标 ROLE-03=✅，但 PC-05/PC-02 按钮前置条件只写"当前用户=publisher_id"，无管理员分支。
- **H-5c 死锁**：管理员下架内容后，原发布者按"仅本人下架内容可由本人重新上架"规则无法恢复，且页面无管理员恢复入口。
- **对应决策**：D5（谁能恢复需拍板）；总数与缺失项我可直接在终稿修正。

### H-6 需求缺 deadline 字段，"超期自动关闭"无计时基准 【两方：架构+后端】
- 需求状态机有"超期自动关闭"转移、架构图/原型承诺"截止时间/截止倒计时"，但 `OpportunityRequest` ER 与 PC-06 字段字典均无任何 deadline 字段——该状态转移无法确定触发时机。与 H-2a 强相关。
- **对应决策**：D2

### H-7 关键指标无基线、北极星缺失 【单方：PM，但影响验收】
- research 明确 Business Outcome / Product Outcome / 成功度量全部"待定/待测量"；总纲 §2.2 五项关键指标（DAU/MAU、发布量、响应时效、采纳率、触达率）除触达率(48h≥80%)外均无基线、无目标值。运营红线阈值（如发布量<20 告警）缺业务依据。
- **影响**：上线后无法判定项目成败，无法验证 ASMP 假设。
- **对应决策**：D6

### H-8 页面 ID 冲突（一号两页）【四方】
- PC-15 = 通知偏好设置(MOD-03,P2) + 公告发布(MOD-05,P1)；PC-20 = 工具助手 + 情报中心。两组各有两个真实 HTML 共用同一 PAGE-ID，路由/权限/埋点必然冲突。
- **处理**：文档缺陷，终稿直接重编号（如公告→PC-24、情报中心→PC-25），无需你拍板。

### H-9 接口契约不成体系 【两方：后端+架构】
- 路径基名不一致（发布需求 `POST /opportunity-requests` vs 采纳/关闭 `/requests/{id}`）；`.../publish` 同端点承载"草稿首发"和"重新上架"两种语义；无统一分页参数名、排序、鉴权头、错误码/错误格式、幂等规范。
- **处理**：需出一份统一 API 契约（终稿补，见 D-tech）。

### H-10 数据模型缺口（建表即暴露）【两方：架构+后端】
- 无 `version` 乐观锁字段（并发编辑/采纳/关闭无原子保证）；`Interaction` 无 `(user_id,target_type,target_id,type)` 唯一约束（重复点赞/收藏、计数错乱）；无 ViewLog（24h 浏览去重无落库）；`OpportunityRequest` 缺 `is_deleted`；Notification 缺 `batch_id`（已读率无法按推送批次聚合计算）；采纳双写（`OpportunityRequest.adopted_response_id` + `SolutionResponse.is_adopted`）双真相源；两份 ER（business-process §C vs IA §ER）不一致未收敛。
- **处理**：终稿数据模型章节统一修正。

## 3. 原型与 PRD 的具体不符（前端逐页比对，终稿以决策为准后统一）

- **富文本能力**：PC-03 原型用 Quill 但工具栏无图片/表格，PC-05 提交方案用纯 TextArea，均与 PRD"支持图片粘贴/上传+表格"不符；全站富文本库选型不统一。
- **评论**：PRD"无限层级+软删占位" vs 原型"两级扁平+硬删除"。
- **PC-13 SLA**：原型无升级等级筛选/列/L0-L3 枚举、无手动升级、无升级链时间线可视化；硬编码分钟阈值 vs PRD"剩余<30%"。
- **PC-12 看板**：原型缺部门活跃度柱状图、告警列表、已读率卡、阈值标色；且改成了前端埋点 Web 分析(UV/PV)，与 PRD"业务库直连聚合"取数架构不符；图表为手写 SVG（正式改 ECharts/G2 需重写）。
- **需求发布权限**：PC-04/PC-06 原型标注"销售和产品均可发布需求"，与矩阵"ROLE-01 专属"冲突（见 D5b）。
- **多角色/端口切换**：原型对销售用户也无条件显示"切换到运营中心"，与三角色互斥专精矛盾（见 D5c）。
- **i18n**：中/EN 按钮无 onclick，无任何英文页面，UGC/枚举/分类名/模板多语言方案未交代（见 D-tech）。
- **响应式**：总纲"本期不做移动端"，但 shell 内置完整移动端适配、index 标题写"响应式 Web 端"——矛盾（见 D-tech）。
- **大量原型独有功能无 PRD 依据**（关注/分享/复制为新方案/@mention/过期横幅/相关推荐/代发布/统计卡/快捷 Tab/邀请回答进度/审核页代发代编等）——需逐项决策纳入/降级/移除。
