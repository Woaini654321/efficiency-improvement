# 商机平台 · v1.0 → v2.0 变更说明台账

> **用途**：逐项列出评审报告《跨岗位交叉核对》中 H-1~H-10（高危）、M-1~M-15（中危）问题的"问题 → 本次处理方式 → 对应决策/落地文件"，供读者一眼看清 v2 相对 v1 改了什么。
> **施工基准**：《决策纪要与修正基线》D1~D9 + 技术默认项 + 可自动修正项。
> **落地文件**：`00_项目总纲_v2.md`（简称 00）、`01_全局规约手册_v2.md`（简称 01）。

## 一、高危问题（P0 blocker）处理台账

| 编号 | 问题（v1 现象） | 本次处理方式（v2） | 决策 | 落地位置 |
|:-:|------|------|:-:|------|
| **H-1** | 范围失控：讨论区/情报中心/工具助手/批量任务/吐槽墙/会议/我的任务等六大功能群"在范围内但零需求定义"，约 40% 原型无规格 | v1.0 收敛为**仅 6 模块 42 FEAT** 核心闭环；上述功能群全部移出至"下期 Backlog"；唯一保留公告（轻量运营 P1） | D1 | 00 §2.3、§2.4、§4.1（下期 Backlog 灰显） |
| **H-2** | SLA 体系多重冲突：计时模型二选一未定(H-2a)、urgency/priority 双编码未映射(H-2b)、SSOT 缺解决时限(H-2c)、升级链模型三处冲突(H-2d)、首响即停 vs Collecting 仍倒计时矛盾(H-2e)、升级人无数据源(H-2f) | 建立 **SLA 单一模型**唯一真理源：①按 urgency 自动计时，删除用户自设 deadline 与 DatePicker；②urgency 存储、priority 派生，明确映射 critical=P0/urgent=P1/normal=P2；③本期仅监控首响(2h/4h/24h)，解决时限下期；④升级链固定三级 L1→L2→L3，间隔=首响时限；⑤进入 Collecting 即 responded 停计时停升级，倒计时仅 Pending 显示(修正 BR-018)；⑥升级人从组织/产品线映射解析 | D2、D3 | 01 §2（全节）、§10.2 |
| **H-3** | 组织结构（部门/产品线/负责人）未建模，数据隔离/SLA 升级/邀请定向/可见范围四项能力悬空 | 新增 **Department（SSO 同步）/ProductLine/ProductLineMember（运营维护）** 实体与组织结构模型章节；给出 SLA 升级人解析规则；新增产品线维护页 PC-26 | D3 | 01 §3、§4.1~4.2；00 §2.3、PC-26 |

## 三、v2 关键取值速查（决策纪要收口值，不得改动）

| 主题 | 收口值 |
|------|------|
| v1.0 范围 | 6 模块 / 42 FEAT |
| SLA 首响 | 特急 2h / 紧急 4h / 普通 24h（自然时，UTC+8，起点=created_at） |
| urgency↔priority | critical=P0 / urgent=P1 / normal=P2（urgency 存储，priority 派生） |
| 升级链 | 固定三级 L1(产品线负责人)→L2(产品管理部)→L3(管理层)，间隔=首响时限 |
| 超期自动关闭 | 待响应超时后 7 天 |
| 评论层级 | 最多 2 级 |
| 强制确认 | ROLE-02 手动勾选触发，ROLE-03 监控 |
| 冷启动订阅 | 默认本部门相关分类；同类 10 分钟合并 |
| 下架恢复 | 谁下架谁恢复 |
| 需求发布 | 仅 ROLE-01 |
| 角色 | 单人单角色；运营入口仅 ROLE-03 |
| 北极星 | 需求响应闭环率（分阶段基线机制） |
| 自动保存 | 60s |
| 分页 | 列表 20 / 卡片 12 / 最大 100 |
| 保留期 | 审计 365 / 软删 180 / 通知 90 |
| 附件白名单 | pdf/doc/docx/xls/xlsx/ppt/pptx/jpg/png/zip |
| 并发 | 峰值 200 用户（用户 ~1000） |
| 技术栈 | ES/OpenSearch + RabbitMQ(outbox 最终一致) + ECharts(业务库聚合) + 统一富文本(OSS 图片) |
| 页面重编号 | 公告发布 PC-24 / 公告详情 PC-25 / 产品线维护 PC-26 / PC-15 仅通知偏好 |
