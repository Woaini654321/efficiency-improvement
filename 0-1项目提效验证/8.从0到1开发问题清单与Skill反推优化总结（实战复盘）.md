## 一、开发全程问题与解决总结

按阶段分组，共 25 个典型问题。「固化去向」指该经验最终沉淀到哪一层（记忆 / CLAUDE.md 规则 / skill 流程 / 门禁）。

### 阶段一：环境与脚手架（0→骨架）

| # | 问题 | 现象 | 根因 | 解决方案 | 固化去向 |
|---|------|------|------|----------|----------|

| 2 | 前端门禁永久 FAIL「文件名含中文」 | self-check 无法通过 | 门禁拿页面**绝对路径**匹配中文，项目路径含「大模型提效」 | 项目根目录改纯 ASCII（→dmx） | 记忆 + CLAUDE.md §5 |
| 3 | 改名目录后前端全挂 | `apps/*/node_modules` 变空、`q-cli-run` 丢失 | pnpm workspace junction 在 Windows 改名后断链 | 根目录重跑 `pnpm install` 秒级修复 | 记忆 + CLAUDE.md §9 |

## 二、反推 Skill/框架的问题与优化建议

开发过程反向暴露出 skill 与框架工具链本身的 14 个改进点。「已做」= 本项目内已落地的对策；「建议」= 可回馈框架组/skill 维护者的优化。

| # | Skill/工具 | 发现的问题 | 影响 | 本项目对策（已做） | 优化建议（可回馈） |
|---|-----------|-----------|------|-------------------|-------------------|
| 1 | vue-coding self-check | 中文检测拿**绝对路径**匹配，项目路径含中文即永久 FAIL，报错信息不指向根因 | 排查半天以为是代码问题 | 项目迁纯 ASCII 路径 | 只扫项目内**相对路径**；FAIL 消息直接提示「项目路径含中文」 |
| 2 | vue-coding env 机制 | env 白名单是硬红线但初期无醒目警示，改错后白屏且报错与 env 无关联 | 全站 404 + 白屏，代价最大的一次事故 | 白名单写进 CLAUDE.md §4；env.lock 哈希门禁 | skill 在 env 被改时给出「白名单 + 一键还原命令」而非只报 hash 不符 |
| 3 | AIRequestGuard（@ai-request-guard/core） | 全局 `mode:'mock'` 分支对 viewSchema 二次跑 adapter，与 `pickBySchema` 的输入要求**自相矛盾**，snake→camel 场景无解 | mock 页面数据全空，排查成本高 | 弃全局 mock，request 层 `mockRequest()` 模式（改 19 个 api 文件） | 库内修复该分支；或官方 mock 规范改推 request 层注入 DTO 模式 |
| 4 | q-cli 脚手架模板 | `main.ts` 硬 import `virtual:.../report-sink`（仅 dev 存在），生产构建必炸 | build 无法通过 | dev-gate 动态 import | 模板出厂即带 dev-gate |
| 5 | @q-web-plugin/* 发包 | 随包发布 `.ts` 源码，下游 vue-tsc 连带检查报错（skipLibCheck 救不了） | typecheck 假红 | shims.d.ts + pnpm patch | 发布物只带 `.d.ts` 产物 |
| 6 | self-check 规则文档 | 隐性禁令（a-table/a-upload/a-input-number v-model/内联色/dataIndex）文档未全列，靠 FAIL 反推 | 并行子代理批量踩同一坑 | 坑位沉淀进 `subagent-baseline.md` 作 SSOT 下发 | error-catalog 补全禁令清单及替代写法 |
| 7 | vue-coding reference 机制 | 权威文档默认无本地缓存，不 sync 就只能凭开源经验瞎猜 | 早期走弯路的总根源 | 入场必跑 Step 0（reference-sync 幂等）固化为纪律 | （skill 新版已内置 Step 0，保持） |
| 8 | Request 签名发现路径 | `@q-web-plugin/request` 无 `.d.ts` 只转发实例，真签名藏在 `apps/*/node_modules/@q-cli/libs`，按 REST 惯例臆造必错 | API 幻觉（如不存在的 DELETEURL） | 写第一个 api 前必读真实 `Request.d.ts` | skill 文档显式给签名文件路径（新版已加，保持） |
| 9 | clean-demo.mjs | 清单不全，漏删 4 个 demo 页触发门禁 FAIL | 初始化多一步排查 | 手删 | 补全清单，清完自动跑一次 self-check |
| 10 | java-coding skill | 无「离线单测桩 MyBatis-Plus 链式调用」配方，Mockito 三连坑全靠现踩 | 272 条测试的成本大头 | 沉淀三件套配方（建链→stub、any(Wrapper.class)、initTableInfo 预热） | skill 增补离线测试配方章节；提供测试基类/工具类 |
| 11 | 框架能力文档 | 乐观锁拦截器未装、UAA roles 不可作业务角色、无业务审计中台——文档均未说明，需反编译取证 | 架构决策反复 | 反编译确认后固化为「后端硬约定」记忆 | 框架文档补「能力边界」清单（有什么/没什么/要自建什么） |
| 12 | QMonoX 序列化默认值 | Long→String 全局转换、NON_NULL 丢 key、LocalDateTime ISO 格式——三者都与前端契约相冲 | 前端 NaN/字段消失/total 变串 | VO 级注解收口 + adapter Number() 兜底 | starter 提供一套「前端友好」的默认序列化配置 |
| 13 | 路由 yaml 热更新 | `dir.meta.yaml` 不参与 HMR，touch 无效，无任何提示 | 改菜单排序以为没生效 | 重启 dev server | vite 插件把 yaml 纳入 watch，或文档注明「改 yaml 须重启」 |
| 14 | QUI 登录组件（记住密码） | "记住密码"依赖 `crypto.subtle`，而它仅在安全上下文（https/localhost）可用；http+IP 访问时登录直接抛错中断，且**无用户可见提示** | 团队成员经 IP 试用 dev 站点集体无法登录，须 F12 才能定位 | 告知不勾"记住密码"；或 chrome flag 把该源列为安全上下文 | 组件做特性检测降级：无 crypto.subtle 时自动禁用"记住密码"并给出提示，不阻断登录主流程 |

## 三、一句话规律（复盘收束）

1. **黑盒框架的正确姿势不是读懂它，而是"skill + 门禁 + 真实签名"三源取证**——猜测是一切事故（env 白屏、API 幻觉、mock 全空）的共同根因。
2. **问题只有被固化才算解决**：本项目每个坑都走完「解决 → 记忆 → CLAUDE.md 规则 → 子代理提示词/门禁」的四级闭环，同一坑没有踩第三次。
3. **评审要评"没写出来的部分"**：最大的 P0（方案闭环双缺）恰恰藏在没有 api 文件的页面级 mock 里——盘点要按前端 url 字面量对账，不能按文件清单。
