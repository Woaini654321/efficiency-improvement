## 写在前面：这份文档是什么、怎么读

这是一次真实项目的**全程踩坑复盘**：用 Claude Code（AI 编程助手）+ 公司 Quectel-code 企业框架，**正式开发只用了两天（07-18 开工、07-19 完工）**，从零做出一个可演示的完整系统；过程中一共踩了 25 个坑，反过来也暴露了公司 skill / 框架工具链本身的 14 个可改进点。

如果你没接触过这套 AI + 企业框架的开发方式，**建议先看下面的「背景与名词」**，否则表格里的术语会看不懂；熟悉的同事可以直接跳到问题清单。

## 项目背景与时间线

**项目是什么**：销售商机互助平台——销售把商机需求发上来，各产品线响应提交方案，系统按紧急度自动 SLA 计时催办，配套看板统计、公告、吐槽墙等模块。最终规模：前端 30 个页面、后端 77 个接口、数据库 19 张表。

**怎么做的**：全程由 Claude Code 驱动开发，人负责提需求、做决策、验收；AI 按公司配套的 skill（vue-coding 管前端、java-coding 管后端）的规范写代码。这也是本次复盘的核心命题——**AI + 企业黑盒框架**这条路到底会踩哪些坑、怎么系统性地避免。

**时间线**（正式开发只有 07-18、07-19 两天，前后分别是准备期和交付后事项）：

| 日期 | 阶段 | 里程碑 |
|------|------|--------|
| 07-18 | **开工 · Day 1** |  前后端脚手架搭好、跑通（环境坑集中在这天），对 PM 的 30 页原型做四角色（PM/架构/前后端）交叉评审，定稿 PRD v2.0 |
| 07-19 | **完工 · Day 2** | 前端 30 页全量实现 → 二轮高保真对齐原型 → 契约收口 → 数据库 19 表 → 77 接口从假数据切换到真实后端（"切真"）→ 五角色评审抓出 P0 并修复 |
| 07-20 | 交付后 | 演示就绪 |
| 07-21 | 交付后 | 内网分享给同事试用（又暴露 2 个新坑，即 #24、#25） |

## 背景与名词（不熟悉这套工具链的同事请先读这里）

| 名词 | 大白话解释 |
|------|-----------|
| **Claude Code** | Anthropic 的 AI 编程助手，能直接读写项目代码、执行命令。本项目里它是"主力开发"，人做需求方和验收方 |
| **Skill** | 给 AI 预置的公司级"操作手册"（vue-coding 前端 / java-coding 后端）。AI 按手册的步骤和规范干活，而不是凭开源经验自由发挥 |
| **门禁 / self-check** | skill 自带的代码检查脚本。写完代码必须跑，报 FAIL 就不允许交付；前端 `pnpm dev` 启动前也会强制跑一遍，不过就起不来 |
| **env 白名单** | 前端的环境变量文件被框架当"黑盒"管理：整个文件里只有 1 个变量允许改，其余全部禁改，改了就出严重事故（见 #6） |
| **mock / 切真** | 后端没好之前，前端先用写死的假数据（mock）把页面全部跑通；后端就绪后把假数据切换成真实接口，叫"切真" |
| **DTO / adapter** | 接口返回的原始数据叫 DTO；页面不直接用它，中间有一层转换函数（adapter）把它转成页面好用的格式。这层转换只能做一次，做两次数据就废了（见 #10） |
| **子代理** | 主 AI 派出的并行"分身"，每个分身独立负责一个模块，最后由主 AI 统一合并、跑门禁验收。是大批量页面/接口的提效关键 |
| **竖切** | 一个功能从数据库→后端接口→前端页面一口气打穿，而不是"先写完所有后端再写所有前端"。好处是每个功能随时可验证 |
| **SSOT（单一事实来源）** | 同一份数据/枚举只在一个地方定义，其它地方全部引用它，禁止复制副本。否则多处拼写不一致迟早打架（见 #15） |
| **固化** | 把踩过的坑变成机制：写进 AI 的长期记忆、项目规则文件（CLAUDE.md）、子代理提示词、门禁脚本。目标是同一个坑不踩第二次 |
| **UAA / SSO** | 公司统一登录认证中心。框架自带对接，业务代码一行登录逻辑都不用写 |
| **安全上下文** | 浏览器的安全机制：部分敏感能力（如加密 API）只在 https 或 localhost 下开放；用 `http://IP:端口` 访问时这些能力直接不存在（见 #25） |

## 一、开发全程问题与解决总结

按阶段分组，共 25 个典型问题。最后一列「固化去向」指这条经验最终沉淀到哪一层（记忆 / CLAUDE.md 规则 / skill 流程 / 门禁），这是本方法论的核心——**问题只有被固化才算解决**。

赶时间的话，先看代价最大、最有代表性的 5 个：**#6（改错一个配置文件全站白屏）、#10（假数据全空的框架级 bug）、#14（AI 首轮页面"不像原型"）、#22（评审抓出的 P0 功能缺失）、#25（同事登不上、自己一切正常）**。

### 阶段一：环境与脚手架（0→骨架）

这个阶段的坑全部和"环境"有关：Windows、内网代理、中文路径、pnpm 的目录链接机制。特点是**报错信息和真实原因相距很远**，没经验的话每个都能耗掉半天。

| # | 问题 | 当时的现象 | 真正原因 | 解决方案 | 固化去向 |
|---|------|-----------|---------|----------|----------|
| 1 | 后端安全组件一启动就崩 | 代码编译能通过，但启动时报 `NoClassDefFoundError`（类找不到） | 框架的 security-starter 内部引用了数据库组件（MyBatis-Plus）的类，两者是隐性绑定关系，但文档没写 | 引入 security 组件时必须同时引入 mysql 组件，两者成对出现 | CLAUDE.md §3 红线 |
| 2 | 前端门禁永远不通过，报"文件名含中文" | self-check 怎么跑都 FAIL，但项目里根本没有中文文件名 | 门禁脚本拿**完整绝对路径**做中文检测，而项目放在名为「大模型提效」的目录下——路径里的中文被误判成文件名中文 | 把项目根目录改成纯英文（→dmx） | 记忆 + CLAUDE.md §5 |
| 3 | 目录改名后前端整个挂掉 | `node_modules` 变空、启动命令找不到 | pnpm 在 Windows 上用"目录链接"（junction）组织依赖，目录一改名链接全部断裂 | 在根目录重跑一次 `pnpm install`，几秒修复 | 记忆 + CLAUDE.md §9 |
| 4 | 清理示例页后门禁仍报错 | 官方清理脚本跑完，self-check 还是 FAIL | 脚手架自带的清理脚本清单不全，漏删了 4 个示例页面 | 手动删除残留页面 | 记忆（待回馈框架组） |
| 5 | 内网 git 仓库拉不下来 / 报 401 | clone 和规范文档同步全部失败 | 本机配了外部代理，把内网域名的请求也带出去了；且访问令牌没有注入 | 环境变量 `NO_PROXY` 加上内网域名（让内网请求不走代理）+ 令牌写进 git 全局配置 | 记忆 |

### 阶段二：前端框架黑盒（骨架→全量页面）

QMonoX 前端框架是"黑盒"：不该动的地方碰不得，该有的能力（图表库、日期库）又不许自己加依赖。这个阶段的坑核心都是**边界在哪里没人告诉你，踩了才知道**——其中 #6 是整个项目代价最大的一次事故。

| # | 问题 | 当时的现象 | 真正原因 | 解决方案 | 固化去向 |
|---|------|-----------|---------|----------|----------|
| 6 | **改了环境变量 → 应用白屏 + 全站 404**（最贵一课） | 想做"不走 SSO 的账密登录"，改了 env 文件里几个变量，结果整站崩溃，报错信息完全不指向 env | env 文件是框架黑盒，有严格白名单：整个文件只有 `VITE_API_DEFAULT_SERVICE_BASE_URL` 一个变量允许改。而且框架本身已内置 SSO 登录，根本不需要自己搞 | 用 git 把 env 还原到初始状态，再用哈希校验门禁确认恢复；白名单写进项目规则永久生效 | CLAUDE.md §4 硬红线 + 反馈记忆 |
| 7 | 新项目所有页面都是 404 | 随便访问哪个路由都进 404 页 | 框架的路由生成器有个隐性规则：项目里没有任何一个"正式业务页"时，会**静默丢弃全部路由**——不报错、不提示 | 按 skill 的建模块流程创建第一个业务页即可，不要去怀疑路由配置或 env | CLAUDE.md §5 |
| 8 | 生产构建失败（开发模式一切正常） | build 卡在 `virtual:ai-request-guard/report-sink` 这个模块上 | 脚手架模板的 `main.ts` 写死引入了这个虚拟模块，而它**只在开发模式存在**，生产构建时解析不到 | 把引入改成"仅开发模式才动态加载"的条件写法，生产构建时被自动裁掉 | 记忆（待回馈脚手架模板） |
| 9 | 类型检查被框架自己的包搞红 | 我们的代码没问题，报错全来自框架包内部 | 框架包发布时把源码一起发出来了，类型检查工具会连带检查它们，且常规的跳过手段（skipLibCheck）对源码无效 | 补类型声明文件绕过 + 用 pnpm patch 给框架包打小补丁 | 记忆 |
| 10 | **假数据模式页面全空**（看板/吐槽墙一个数字都没有） | 图表、列表、统计卡全部显示 0 或空 | 框架请求库的全局 mock 模式有个自身矛盾的 bug：对同一份数据**跑了两次 adapter 转换**，第二次转换把数据全部转丢了（4 个场景运行时复现证实） | 放弃框架的全局 mock，在请求层自建一个轻量的假数据开关——好处是后端就绪后关掉开关就是真实调用，一键切真 | 记忆 + 后续一键切真的基础 |
| 11 | 日期库 dayjs 用不了 | import 就报错，而项目红线禁止自己加依赖 | dayjs 没有被框架显式提供给业务代码 | 日期组件改用"直接输出字符串"的属性，日期比较用原生字符串比较（`YYYY-MM-DD` 格式天然可比大小） | 记忆 + 子代理基线 |
| 12 | 没有图表库、没有富文本组件 | 原型里有折线图、饼图、富文本编辑器，框架不提供，又禁止加依赖 | 框架黑盒 + 禁加依赖的双重约束 | 图表用纯 SVG/CSS 手绘（折线用 polyline、饼图用 CSS 渐变）；富文本用浏览器原生的可编辑区域自建 | 记忆 + 共享组件沉淀 |
| 13 | 隐性门禁连环 FAIL | 一批常规写法（antd 的表格、上传、数字输入框、行内颜色）被门禁拒绝，而文档没写这些禁令 | 门禁脚本的禁令范围比规范文档写出来的更广，只能靠 FAIL 反推 | 换用框架指定组件（QBigTable/QUpload）、改用受控写法、颜色走设计变量 | 子代理基线 SSOT |
| 14 | **首轮全量生成的页面"不像原型"** | 30 页是做出来了，但用户看完的反馈是：页面少了、数据丢了、不美观 | 多个子代理并行生成时，各自把复杂布局（卡片流、看板）退化成了最保守的"表格+表单" | 二轮高保真重做：先派 5 路子代理逐页比对原型找差距 → 主 AI 统一定组件范式和样板页 → 8 路子代理照样板镜像重做 → 最后集中跑门禁 | 方法论文档第 5 篇 |

### 阶段三：契约与数据库（页面→真库）

前端页面全部跑通后、建数据库之前，有一个容易被跳过的关键动作：**把前后端对数据的"约定"（契约）统一掉**。这个阶段的坑说明：假数据阶段各写各的，建库时一定打架；而框架的序列化默认行为也和前端预期有好几处冲突。

| # | 问题 | 当时的现象 | 真正原因 | 解决方案 | 固化去向 |
|---|------|-----------|---------|----------|----------|
| 15 | 前后端枚举/状态值打架 | 同一个状态在不同页面拼写不一致（expired vs archived、done vs completed） | 假数据阶段各页面各写各的，没有唯一定义处 | 建库前做一轮"契约收口"：字典表作为枚举的唯一来源，全项目扫一遍统一 | 方法论文档第 6 篇 |
| 16 | 序列化三连坑 | ①分页总数变成字符串；②值为 null 的字段整个消失；③日期到前端显示 NaN | 框架的 JSON 序列化默认行为：长整型统一转字符串（防前端精度丢失，但分页总数也被误伤）、null 字段直接不输出、日期用 ISO 格式而前端期待另一种 | 前端转换层对总数强转数字兜底；后端在出参对象上加注解精确控制格式 | 独立记忆（序列化契约三坑） |
| 17 | 框架的审计/乐观锁能力与预期不符 | 文档没写清楚框架到底提供了什么，代码又是黑盒 | 反编译框架 jar 包取证后确认：乐观锁拦截器框架**没有装**（要自己补）；统一认证中心的角色**不能**当业务角色用 | 自己注册乐观锁拦截器；业务角色存本地用户表自行管理 | 独立记忆（后端硬约定） |

### 阶段四：竖切联调与切真（真库→演示就绪）

77 个接口从假数据切换到真实后端，采用"竖切"打法：每个功能从库到接口到页面一次打穿，10 路子代理并行写码。这个阶段最重要的收获是 **#22——五角色评审抓出了一个真正的 P0 级功能缺失**，而它恰恰藏在最不容易被盘点到的地方。

| # | 问题 | 当时的现象 | 真正原因 | 解决方案 | 固化去向 |
|---|------|-----------|---------|----------|----------|
| 18 | 子代理写的代码两次炸编译 | Java 文档注释提前终止，编译失败 | 子代理在中文注释里写了字面的 `*/` 字符（比如"create*/update*"这种缩写），Java 把它当成了注释结束符 | 修掉注释；这条规则写进之后所有子代理的提示词 | 子代理提示词模板 |
| 19 | 需求详情页打开就 500 | 已采纳的需求一点开就报服务器错误 | 种子数据（预置的演示数据）的 JSON 结构和 Java 实体类声明的类型不匹配：实体要"字符串列表"，种子塞的是"对象数组"，反序列化直接炸 | 修正 7 行种子数据；立规矩：种子 JSON 的结构必须和实体类型一一对应 | 记忆 |
| 20 | 集成测试和演示数据互相删对方 | 一个测试时红时绿，很诡异 | 测试把一条真实的演示种子数据当成了自己的"沙箱"，测试清理时把演示数据删了，演示数据重建时又干扰测试 | 给测试划一段专用的保留 ID 区间，和演示数据彻底隔离，并在注释里写明警示 | 测试代码内固化 |
| 21 | 单元测试模拟数据库链式调用连环报错 | Mockito（Java 测试模拟工具）报出一连串晦涩错误 | MyBatis-Plus 的链式查询写法在离线测试里怎么模拟，业界没有现成配方，三个坑全靠现踩 | 沉淀出三件套配方（建链顺序、重载消歧、表信息预热），后续项目直接复用 | 记忆（可复用配方） |
| 22 | **评审抓出 P0：核心闭环"方案匹配"前后端双缺** | 需求详情页的"提交方案/采纳方案"看着能点，实际全是假的——后端没有接口，前端也没有真实调用 | 这个功能是"页面级假数据"（假数据直接写在页面里，没有独立的接口文件），所以做接口盘点时（当时盘出 68 个端点）它**根本不在清单上**——这是假数据隐藏问题的第三种形态 | 五角色评审（PM/架构/前端/后端/测试各一个视角）把它揪了出来，前后端补齐整条链路并走通，接口终态达到 77 个 | 方法论文档第 7 篇 |
| 23 | 切真之后反而觉得"数据变少了" | 好几个模块比假数据阶段看着空 | 假数据只存在于前端代码里，不会自动变成数据库数据 | 编写 224 行核心演示种子数据（专用 ID 段、可重复执行），再做 15 项逐模块数据走查 | db/seed 落库 |

### 阶段五：演示与内网分享（就绪→给同事用）

系统做完≠别人能用起来。把 dev 站点分享给同事试用的当天又踩了 2 个坑，共同特点是：**在自己机器上永远复现不了，换个人、换台机器才暴露**。

| # | 问题 | 当时的现象 | 真正原因 | 解决方案 | 固化去向 |
|---|------|-----------|---------|----------|----------|
| 24 | 改菜单排序不生效 | 改了菜单配置文件（dir.meta.yaml），刷新没变化，怎么触发都没用 | 这类路由配置文件只在开发服务器**启动时**读取一次，不参与热更新，且没有任何提示 | 重启开发服务器即生效 | 记忆 |
| 25 | 同事登录报错/毫无反应，我自己一切正常 | 我用 `localhost` 访问一切正常；同事用 `http://IP:8080` 访问，勾了"记住密码"点登录就抛 `crypto.subtle is not available`，页面上却没有任何提示 | 浏览器的加密 API 只在**安全上下文**（https 或 localhost）下存在；框架登录组件的"记住密码"功能依赖它加密存储账密——同事走 http+IP 访问，这个 API 根本不存在，登录流程直接中断 | 不勾"记住密码"即可正常登录（登录本身不依赖加密 API）；确需此功能可在 `chrome://flags` 把该地址列为安全来源；根治是部署到 https 环境 | 记忆（阶段五新增） |

## 二、反推 Skill/框架的问题与优化建议

上面 25 个坑里，有相当一部分的根因不在业务代码，而在 skill 和框架工具链本身。这一节把它们反向整理成 14 条反馈：**「本项目对策」是我们当时已经落地的临时解法，「优化建议」是可以回馈给框架组/skill 维护者的根治方案。** 这也是这次实战对公司工具链最有价值的产出之一——工具的问题只有被真实项目碾过一遍才会全部现形。

| # | Skill/工具 | 发现的问题 | 造成的影响 | 本项目对策（已做） | 优化建议（可回馈） |
|---|-----------|-----------|-----------|-------------------|-------------------|
| 1 | vue-coding 门禁脚本 | 中文检测拿**完整绝对路径**匹配，项目放在中文目录下就永久 FAIL，且报错信息不指向真实原因 | 排查半天，一直以为是自己代码的问题 | 项目迁到纯英文路径 | 只扫项目内**相对路径**；FAIL 消息直接提示"项目路径含中文" |
| 2 | vue-coding env 机制 | env 白名单是硬红线，但初次接触时没有醒目警示；改错之后白屏，报错还和 env 毫无关联 | 全站 404 + 白屏，整个项目代价最大的一次事故 | 白名单写进项目规则；加了 env 哈希校验门禁 | env 被改动时给出"白名单清单 + 一键还原命令"，而不是只报哈希不符 |
| 3 | 框架请求库（AIRequestGuard） | 全局假数据模式对同一份数据跑两次转换，与库自身另一处的输入要求**自相矛盾**，特定场景下无解 | 假数据页面全空，排查成本极高 | 弃用全局假数据模式，请求层自建开关（改了 19 个接口文件） | 库内修复该分支；或官方假数据规范改推荐请求层注入的模式 |
| 4 | 脚手架模板 | 入口文件 `main.ts` 写死引入 `virtual:ai-request-guard/report-sink`（该虚拟模块只在开发模式存在），生产构建必炸 | 生产构建无法通过 | 改成仅开发模式动态加载 | 模板出厂就带条件加载写法 |
| 5 | 框架前端包的发布方式 | 把源码连同产物一起发布，下游类型检查被连带报错，常规跳过手段无效 | 类型检查出现"假红"，真问题被淹没 | 补类型声明 + pnpm patch 打补丁 | 发布物只带编译产物和类型声明，不带源码 |
| 6 | 门禁规则文档 | 一批隐性禁令（禁用某些 antd 组件、禁行内颜色等）文档没有全列，只能靠 FAIL 反推。禁令来源=vue-coding skill 的 self-check 门禁脚本（框架组随 skill 下发，意图是统一到公司组件体系），问题在于**脚本实际检查的规则集比文档公开的更广** | 并行子代理批量踩同一个坑 | 把坑位沉淀成子代理基线文档统一下发 | 错误速查手册补全禁令清单及每条的替代写法 |
| 7 | vue-coding 规范文档机制 | 权威规范默认没有本地缓存，不主动同步就只能凭开源经验瞎猜 | 早期走弯路的总根源 | 把"入场必先同步规范"固化为纪律 | （skill 新版已内置该步骤，保持） |
| 8 | 请求工具的真实方法签名藏得太深 | 前端调接口都走框架的 request 工具，但业务代码 import 的那个包（`@q-web-plugin/request`）**没有任何类型定义**——只是转手导出实例，顺着 import 去查必然扑空；真实的方法签名藏在另一个包的深层目录（`apps/web-app/node_modules/@q-cli/libs/dist/request/Request.d.ts`），没人告诉你在哪。查不到就只能按业界 REST 惯例猜，而这套工具的方法命名和惯例并不一致 | AI 凭空臆造出不存在的方法（幻觉），如根本没有的 `DELETEURL`（实际删除就叫 `DELETE`） | 立纪律：写第一个接口文件前，必须先读一遍真实的 `Request.d.ts`，禁止按惯例猜 | skill 文档显式给出签名文件路径（新版已加，保持） |
| 9 | 示例页清理脚本 | 清单不全，漏删 4 个示例页，触发门禁 FAIL | 初始化平白多一步排查 | 手动删除 | 补全清单；清理完自动跑一次门禁自检 |
| 10 | java-coding skill | 后端单元测试要"离线"跑（不连真实数据库，用模拟工具 Mockito 伪造数据库返回值），但框架的数据库操作是 MyBatis-Plus 链式写法（形如 `lambdaQuery().eq(...).list()` 一路点下去的调用链）——这种链怎么模拟，skill 没写、网上也没有现成答案，只能边报错边摸索，连踩三个坑（模拟链要先建好才能定义返回值、方法重载导致模拟不知道匹配哪个、框架内部表信息缓存未初始化直接报错） | 272 条离线测试里，时间成本的大头都耗在这上面 | 摸索出三件套配方沉淀下来（先建链再定义返回值、用 `any(Wrapper.class)` 消除重载歧义、用 `TableInfoHelper.initTableInfo` 预热缓存），后续项目可直接复用 | skill 增补离线测试配方章节；提供测试基类/工具类 |
| 11 | 框架能力文档 | 乐观锁拦截器没装、统一认证的角色不能当业务角色、没有业务审计中台——这些"框架没有什么"文档都没写，只能反编译取证 | 架构决策反复摇摆 | 反编译确认后固化为"后端硬约定" | 框架文档补一份「能力边界」清单：有什么 / 没什么 / 需要自建什么 |
| 12 | 框架序列化默认值 | 长整型全局转字符串、null 字段丢失、日期格式不符——三个默认行为都和前端约定冲突 | 前端出现 NaN / 字段消失 / 数字变字符串 | 出参对象加注解精确控制 + 前端转换层兜底 | 框架提供一套"前端友好"的默认序列化配置 |
| 13 | 路由配置热更新 | 菜单配置 yaml 不参与热更新，改了没反应，也没有任何提示 | 以为改动没写对，反复试 | 重启开发服务器 | 把 yaml 纳入监听自动重启；或文档注明"改 yaml 须重启" |
| 14 | 框架登录组件（记住密码） | "记住密码"依赖浏览器加密 API，而该 API 仅在安全上下文（https/localhost）可用；http+IP 访问时登录直接中断，且**没有任何用户可见的提示** | 团队成员通过 IP 试用 dev 站点集体无法登录，必须打开浏览器开发者工具才能定位 | 告知大家不勾"记住密码"；或用 chrome flag 把该地址列为安全来源 | 组件做特性检测降级：检测不到加密 API 时自动禁用"记住密码"并给出提示，不阻断登录主流程 |

## 三、一句话规律（复盘收束）

1. **黑盒框架的正确姿势不是"读懂它"，而是三源取证：skill 规范 + 门禁反馈 + 真实签名。** 回看整个项目，代价最大的几次事故（env 白屏、API 幻觉、假数据全空）有一个共同根因——在拿不到权威信息的地方靠猜。猜测在黑盒框架面前的命中率低得惊人，而"先把规范同步下来、写码前先读真实签名、写完立刻跑门禁"这三个动作能把猜测挤出流程。

2. **问题只有被固化才算解决。** 修好一个 bug 只是第一步；本项目每个坑都走完了"解决 → 写进记忆 → 升级为项目规则 → 写进子代理提示词/门禁"的四级闭环。效果是可量化的：同一个坑没有被踩过第三次。对团队来说，这份清单本身就是固化的产物——你们不需要再踩一遍这 25 个坑。

3. **评审要评"没写出来的部分"。** 本项目最大的 P0（方案匹配闭环前后端双缺）恰恰藏在没有独立接口文件的页面级假数据里——按文件清单盘点接口永远盘不到它。教训是：盘点要拿前端页面上实际发起的请求地址去对账，而不是拿代码文件清单对账；评审的多角色视角（PM 问"业务闭环在哪"、测试问"这条路径真的通吗"）正是为了照亮这些文件清单照不到的暗处。
