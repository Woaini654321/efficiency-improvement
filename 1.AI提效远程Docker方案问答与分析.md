## 一、问题原文

> 我们公司想要使用 AI 大模型辅助开发进行开发人员提效，当前的方案是使用远程服务器部署 docker，docker 内部安装编译器、claude code、git 等开发工具，大家使用网页和脚本与 docker 进行通信，控制 docker 内的 claude、编译器、git 等开发工具，这样实现了开发环境的统一，并且能共享项目记忆，但是也存在很大的问题，比如安全风险、同时读写同一份文件等等，现在有比较成熟的解决方案嘛？

## 二、分析与解答

### 2.1 总体判断

有成熟方案，而且现有方案的**方向（远程统一环境）本身是对的**——业界这类形态叫 **CDE（Cloud Development Environment，云开发环境）**，已相当成熟。

现有方案的真正问题**不在"远程 Docker"，而在"多人共享同一个容器/同一份工作区"**——这一个设计点改掉，安全和并发写两个大问题同时消解。

### 2.2 核心原则

**共享的是镜像和规范，隔离的是工作区，汇聚靠 git。**

成熟方案没有一家让多人读写同一份文件，标准形态：

```
黄金镜像（编译器+Claude Code+git+代理配置，统一维护）
    │  按需实例化
    ▼
每人/每任务一个独立容器（自己的 clone 或 git worktree）
    │  产出以 MR/PR 形式汇聚
    ▼
GitLab 受保护分支（评审+CI 门禁做串行化）
```

并发冲突由 git 的 merge 机制解决——三十年验证过的并发模型，比任何文件锁方案都成熟。现状的"同时读写同一份文件"问题，本质是**把 git 该干的活让文件系统干了**。

### 2.3 成熟方案选型对比

| 方案 | 形态 | 内网适配 | 结论 |
|---|------|:---:|------|
| **Coder（自托管）** | 开源 CDE 平台：管理员定义模板/镜像，每人一键开自己的容器，浏览器 IDE + SSH 都支持，**内置 AI Agent/Tasks 功能可直接跑 Claude Code** | 高 | **推荐主选**——纯内网部署，对接 GitLab OAuth 做身份，正好替换自研的"网页+脚本"控制面 |
| code-server / JetBrains Gateway | 轻量：每人一容器跑浏览器版 VS Code，Claude Code 装容器里 | 高 | **推荐起步**——最小改造路径 |
| Gitpod 自托管（Ona）/ Eclipse Che | 同类 CDE，按仓库快照秒开工作区 | 中 | 部署较重，备选 |
| GitLab CI + headless agent | `claude -p` 无头模式在 CI 里按任务起容器，产出 MR | 高 | **与 CDE 互补**——适合批量任务（批量迁移、修 lint、巡检） |
| Claude Code on the Web | Anthropic 托管沙箱，每任务独立 VM，产出 PR | 低 | 代码需出内网，多数企业内网场景不适用 |

### 2.4 两个核心痛点的正解

**痛点一：安全风险**

共享容器的致命处：所有人的凭据与代码在同一容器内——任何一人的误操作或 **prompt 注入**，能读取/外带全部人的代码与凭据，爆炸半径是整个团队。改成每人一容器后，再叠加五层标准做法：

| 层 | 措施 |
|---|------|
| 凭据 | 不进镜像：git 用短时效 token（GitLab OAuth/OIDC 下发）；Claude API key 走网关统一注入并计量审计 |
| 网络 | 容器出网走 egress 代理**白名单**（只放行 Nexus、GitLab、LLM 网关），防数据外带 |
| 容器 | rootless 运行 + 不挂 docker.sock；需更强隔离用 gVisor / Kata |
| AI 权限 | Claude Code permission mode 按角色配置（受管 settings 打进镜像），配合操作审计日志 |
| 定责 | 每容器绑定唯一身份，"谁让 AI 干了什么"天然可查 |

**痛点二：共享项目记忆**

不要用共享卷共享 `~/.claude`（那是并发写问题的另一来源）。正确做法是**记忆走 git 版本化**——这套模式已在 sales-lead-hub 项目实战跑通：

| 层级 | 载体 | 说明 |
|---|------|------|
| 项目级 | 仓库内 `CLAUDE.md` / `MEMORY.md` / `AGENTS.md` | clone 即得，天然带评审、历史、回滚 |
| 公司级 | 内网 skill/插件市场（quectel-plugin-marketplace 的 vue-coding / java-coding 即此形态） | 企业框架经验做成 skill，预装进黄金镜像全员复用 |
| 个人级 | 各自容器的私有卷 | 不共享，避免脏写 |

### 2.5 推荐路径（现有投入不推翻）

远程服务器和统一镜像**保留**，做三个替换：

1. "一个共享容器" → **Coder（或先 code-server 起步）管理的每人一容器**；
2. "共享记忆卷" → **仓库内 CLAUDE.md + 内网 skill 插件市场**；
3. "直接写同一份文件" → **每任务 git worktree/clone + MR 汇聚**。

自研网页控制面可以退役——那是 CDE 平台已经做好的部分；精力应花在**黄金镜像、skill 市场、门禁规范**上，那才是别人替代不了的企业资产。
