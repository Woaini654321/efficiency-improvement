# 用 Claude Code 做契约收口与数据库从 0→1（框架黑盒下的建库落地）· 方法论复盘

> 一次真实会话的复盘：**在"前端高保真已交付、黑盒缺陷已排障"之后，把项目继续往下推——先从被压缩的会话里恢复丢失的任务定义续跑；再把跨表分裂的枚举/SSOT/PRD 明文修正在前端一次性"收口"，为建库扫雷；然后从 0→1 落数据库——选型、按 PRD ER 写 DDL、反编译框架 starter 定审计/主键/逻辑删除约定、种子数据、MyBatis-Plus 实体与配置；最后用"架构师 + 后端工程师"双角色评审→取共识→修复→复验编译。**
> 载体仍是「销售商机互助平台」（Quectel-code 企业框架 + QMonoX 前端 + Spring Boot 2.7 + MyBatis-Plus + UAA SSO）。本文是 0→1 系列的**第六篇**，与《AI 全程搭建 0→1 方法论》《Quectel-code AI 开发正确姿势》《用 Claude Code 做 PRD 评审与产品文档方法论》《用 Claude Code 打造项目 AI 配置三件套方法论》《用 Claude Code 排障黑盒缺陷与评审需求还原度方法论》配套——**前五篇讲"做出来、验证对、修好环境"，这一篇讲"把前端契约收口干净，再从零把数据层稳稳建起来"**，让整条 0→1 链路不断档。

## 一、结论先行（TL;DR）

**从"页面能跑"到"库能建"，中间横着一道最贵的坎：契约不收口就建库，等于给自己埋返工。而建库本身在企业黑盒框架下，最大的风险是"按开源惯例猜框架约定"。**

- 🧭 **续跑先恢复任务定义，别猜**：新窗口只知道"继续 A+C"，可 A/C 的确切含义在被压缩的会话里丢了。不臆断——**从会话记录（transcript）里把当时用户的选择抠回来**，锁死 A=契约收口、C=PRD 明文修正，再动手。
- 🧹 **前端先收口，再建库（扫雷）**：枚举跨表分裂（公告 `expired` vs 基线 `archived`）、SSOT 多副本（分类/产品线字典分叉）、PRD 明文点未落地（残留 `deadline`）——**这些在前端一次对齐，后端建表直接照字段落库，避免"建完表再改字段"的雪崩。**
- 🔬 **框架约定靠证据，不靠经验**：审计列名、主键类型、逻辑删除写法，不拍脑袋——**反编译 starter 字节码**看它到底填哪几个字段、`createBy` 填的是不是 `Long`；这连锁定死"主键 BIGINT + 前端 Long→String"。这与上一篇"最小复现钉黑盒"是同一硬核。
- 🧑‍⚖️ **双角色评审→共识→修复→知道何时停**：架构师看 ER 对齐/范式，后端工程师看注解/类型；两方 PASS 后只修**真结构问题**，把结论分成"改结构 / 写入纪律 / 运行时自证"三层；当剩余风险只能运行时验证时，**不再空转评审**。

> 一句话方法论：**契约在最上游对齐（前端收口为建库扫雷）；框架约定用字节码/文档钉死、绝不按开源惯例猜；建库以 PRD ER 为唯一权威、以门禁/编译为验收；真岔路停下让人拍板，双角色评审取共识只修真结构，剩余只能运行时验的就别再照静态一遍。**

## 二、正确的主线流程（可照抄）

```
续跑线（会话压缩后继续任务）
  1. 新窗口先验工具：跑 skill 前置体检 + self-check，确认 vue-coding 恢复可用
  2. 任务定义丢了不猜：从 .claude/projects/*.jsonl 抠出当时的 AskUserQuestion 选项与用户回答
  3. 锁死语义再动手（本次发现"最初4选、后收窄成 A+C"，以最后一次指令为准）

收口线（前端契约/SSOT/PRD 修正，为建库扫雷）
  4. 分层清单：A 建库硬冲突（枚举/SSOT）/ C PRD 明文修正（deadline/频控/软删/拖拽）
  5. 真岔路先问人：枚举二选一、SSOT 收敛"轻/重"——影响建表列值/会级联改表单，交人拍板
  6. 多子代理并行按"文件互斥"切分；共享文件（locale）不并行改 → 回传片段 → 主控中心化合并
  7. 中心化跑门禁到绿（typecheck:all + self-check 0/0），残留全局抽查清零

建库线（数据库从 0→1）
  8. 选型：识别"被框架+基础设施+数据模型三层锁定"→ 讲清为什么没得选（MySQL）
  9. DDL 以 PRD §4 ER + §10 枚举为唯一权威，逐表逐字段逐枚举对齐
  10. 框架约定反编译 starter 拿 ground truth（审计列名/主键 Long/逻辑删除 deleted 'Y'/'N'/禁定义 Config&Handler）
  11. 种子 data.sql（分类树照前端权威 SSOT）+ MyBatis-Plus 实体&配置 → mvn compile 复验绿
  12. 依赖触红线先问（Lombok）→ 同意后再装

评审线（数据库产物双角色把关）
  13. 架构师（ER 对齐/范式/约束）+ 后端工程师（注解/类型/框架约定）并行只读评审
  14. 交叉取共识：只修真结构问题；结论分三层（改结构 / 写入纪律 / 运行时自证）
  15. 判断是否再评审：边际递减 + 剩余只能运行时验 → 停，转去跑起来验
```

## 三、关键方法与技巧复盘（本文精华）

### 技巧 1 · 会话被压缩后，任务定义从"记录"里抠回来，绝不猜
新窗口开局只有一句"继续帮我实现 A+C"，而 A/C 的确切含义在上一段被上下文压缩吃掉了。**错误做法是按字面猜一个 A/C 硬着头皮做**；正确做法是把它当"取证"：直接读 `~/.claude/projects/<项目>/*.jsonl` 会话记录，用一段 Python 把历史里的 `AskUserQuestion` 选项与用户的回答抠出来——一眼定位到当时那道"打磨范围/收口清单"多选题，甚至还原出"用户最初四项全选、后来在某条指令里收窄成 A+C"的演化。**续跑的第一原则：任务定义以记录为准，恢复不了就问用户，绝不猜着往下做**——猜错方向的返工，比花五分钟抠记录贵得多。

### 技巧 2 · 前端先收口，再建库——把契约对齐做在最上游（"建库前扫雷"）
建库最贵的返工是"表建完了才发现字段口径不对"。所以正确顺序是：**在写第一行 DDL 之前，先把前端里所有会影响建表的契约分裂一次收口**——
- **枚举跨表分裂**：公告状态前端用了 `expired`，而 PRD 基线是 `archived`；需求 `status` 在需求模块是 `Pending/…`(PascalCase)、在审核模块却是小写——**同一后端字段两套口径，建库即撞**。
- **SSOT 多副本**：分类/产品线的"选项字典"散在多个 mock 且命名分叉（`5G模组` vs `5G 模组`）；公告实体前后端两套互不相干的数据。
- **PRD 明文点未落地**：残留已删的 `deadline` 字段（违背决策 D2）、催办频控写成 10 秒（应 ≥10 分钟）、评论硬删（应软删占位保留子回复）。

把这些在前端对齐后，**后端 ER/建表直接照收口后的字段落库，一步到位**。方法论：**数据契约在数据流最上游对齐，成本最低；越往下游（建表、联调）改，波及面越大。**

### 技巧 3 · 多子代理并行的正确姿势——"文件互斥"切分 + 共享文件回传片段、主控中心化合并
约 25 个文件的契约级改动，用 5+3 路 subagent 并行。并行不翻车的核心不是"人多"，而是**切分纪律**：
- **按文件互斥分派**：每个 agent 只碰自己模块的文件（公告 / 审核 / 需求 / 分类 / SLA…），彼此零重叠，可放心并发。
- **共享文件绝不并行改**：两份 `locale/*.ts` 几乎每个任务都要碰——**让 agent 一律不直接改 locale，只在报告里回传"locale 片段"，由主控中心化合并**，规避并发写同一文件的冲突。
- **中心化门禁**：所有 agent 只产出、不各自跑全局门禁（会互相干扰）；主控合并后统一跑 `typecheck:all + self-check` 到绿。

这套"并行取证/生产、主控整合裁决"，与上一篇评审的多角色打法同源——**并发提速，串行收口**。

### 技巧 4 · 选型不是选择题——被约束锁定时，讲清"为什么没得选"比罗列优劣有用
"MySQL 还是 PostgreSQL"看似开放，实则**三层锁定，没有选择空间**：① **框架层**——`quectel-code-mysql-starter` 提供 MyBatis-Plus，而 `security-starter` 的审计填充强绑它，换 PG 等于弃用整套企业脚手架；② **基础设施层**——`datasource` 的 JDBC URL 已写死 MySQL 实例、`createDatabaseIfNotExist=true`；③ **数据模型层**——关系型 + JSON 列，PG 的独门优势（复杂 GIS/数组/物化视图/内建全文检索）本项目一个都不触发（要全文检索也是走 `elasticsearch-starter`）。**好的技术回答不是甩一张优劣对比表，而是识别出"这是被约束决定的"，把约束讲透，省掉无谓比较。**

### 技巧 5 · 框架约定靠"字节码/源码"取证，不靠开源惯例猜（本篇最硬核）
建库最容易翻车的地方是"按通用 MyBatis-Plus 经验猜这套企业框架的约定"。审计列名、主键类型、逻辑删除写法，**全部反编译 starter 拿 ground truth**：
- `javap` 反编译 `SecurityMetaObjectHandler`：看到它 `strictInsertFill` 的真实字段名是 `createTime/updateTime/createBy/updateBy`，且 `createBy` 填的是 `getCurrentUserId():Long`——**"当前用户 id 是 Long"这一条，连锁定死了"全表主键 BIGINT + 雪花，前端 ID 全 string 由 Jackson `Long→String` 满足"**。
- `mysql-starter` 规范文档定死两条硬红线：**逻辑删除字段名是 `deleted`、值 `'Y'/'N'`**（不是想当然的 `is_deleted` TINYINT）；**业务侧禁止定义 `MyBatisPlusConfig` 和任何 `MetaObjectHandler`**（会 Bean 冲突）。
- 由此还揪出一个语义 bug：评论 D7 软删是"行保留、内容转占位、留子回复"，与框架 `deleted`"隐藏整行"语义**冲突**——必须拆出独立业务字段 `content_deleted`。

这与上一篇"用最小复现把黑盒真因钉在某一行"是**同一方法论**：**黑盒行为靠证据，不靠直觉。** 甚至诚实到——文档说 starter 的配置"含乐观锁"、但我反编译的那份字节码只见分页拦截器，就把它明确标为"运行时自证点"，而不是假装已验证。

### 技巧 6 · 遇真岔路停下让人拍板——枚举 / 范围 / 依赖
AI 不该在**真正的设计岔路**上擅自定方向。本次三次停下问用户，判据一致：
- **公告枚举**：PRD 两处自相矛盾（`archived` vs `withdrawn`）——**它决定建表那一列的取值**，交人定（选 `archived`）。
- **SSOT 收敛"轻/重"**：值对齐（低风险、观感不变）vs id 级单源重构（**会级联改两个表单的存储值**、观感变、风险高）——交人选（选"轻"）。
- **装不装 Lombok**：触碰"禁未批准新依赖"红线——**先问再装**，同意后才加 pom。

判据可复用：**当答案会改变产物、触碰红线、或 PRD 自相矛盾时，停下问；有惯例默认值的自己定、说明即可。** 别把"该问的"擅自定了，也别把"该定的"都推给用户。

### 技巧 7 · 双角色评审→共识→修复，且把结论分成"能修的"与"只能运行时验的"
数据库产物用两个 lens 并行只读评审——**架构师**（ER 对齐/范式一致/约束落实/索引/SSOT）+ **后端工程师**（MyBatis-Plus 注解/类型映射/框架约定/编译与运行时坑）。两方都给 PASS 后，交叉取共识**只修真结构问题**，并把结论**分三层**处置：
- **改结构（当场修）**：补 `request_product_line` 关联表——原设计里"分类走关联表、邀请产品线却只有 JSON"是唯一的范式不一致，且导致"某产品线被哪些需求邀请"的反查（SLA 升级人解析要用）无法走索引；顺手加 `category` 同级名唯一约束、种子补"产品管理部"。
- **写入纪律（注释即可）**：聚合计数/批次已读率的同事务维护、SLA 升级幂等——不改结构，写进 DDL 注释。
- **运行时自证（静态照不出）**：乐观锁拦截器是否真装、下划线转驼峰开关、雪花 workerId——**只能跑起来验**。

分层的价值：**让"评审结论"变成"可执行的分工"，而不是一堆等价的问题。**

### 技巧 8 · 知道何时停止评审——边际价值递减 + 剩余风险只能运行时验时，别再空转
用户问"还要不要再来一轮多角色评审"，诚实的答案是**不要**，判据三条：① 上一轮双 PASS、本次修复是小增量且 `mvn compile` 复验绿，再照一遍大概率只复述同样几条已归档 Minor；② 剩余没闭合的全是**运行时自证项**，再多轮静态评审也变不成"已验证"；③ 又一轮多代理是实打实的成本换重复结论。**真正降风险的下一步是"跑起来"——起库 + 首个模块读写用例 + 一个 `@Version` 并发用例 + 一个"无权限角色应 403"用例，而不是再静态照一遍。** 知道何时停，和知道何时评审，同样是本事。

## 六、效率与成本的诚实备注

- **成本主要花在四处**：① 从 transcript 恢复任务定义 + 前端约 25 文件契约收口（5+3 路并行 subagent）；② 反编译 starter 字节码定框架约定（一次性、但决定成败）；③ 生成 18 表 DDL + 种子 + 全套实体/Mapper/config；④ 双角色评审（各独立 Opus 上下文并行）。
- **省钱要点**：
  - **上游收口省下游返工**：契约在前端对齐一次，胜过建表后逐字段回改——最贵的是返工，不是收口。
  - **并行生产、主控整合**：子代理 background 并行、墙钟＝最慢一路；共享文件不并行改，避免"合并冲突"这种隐性重做。
  - **一次取证胜过反复猜**：反编译一次 starter，把审计/主键/逻辑删除约定全定死，省掉"猜错约定→编译/运行报错→再改"的多轮。
  - **知道何时停**：评审到"剩余只能运行时验"就停，不为一份重复结论再烧一轮多代理。
  - **坑与决策升格记忆**：框架约定（deleted='Y'、禁定义 Config/Handler、createBy 是 Long）、SSOT 收口口径写进项目记忆，后续会话直接命中。
- **可推广性**：这套"续跑先取证→上游收口→反编译定框架约定→PRD 为权威建库→双角色取共识→知道何时停"的主线，适用于**任何"在企业黑盒框架下、从已有前端契约往下落数据层/后端"**的场景，可作团队 SOP。

## 七、一页纸总纲

> **续跑先从记录里恢复任务定义、绝不猜；契约/枚举/SSOT 在前端最上游一次收口，为建库扫雷；选型识别被三层锁定的必然、把约束讲透；框架约定反编译 starter 用字节码钉死、不按开源惯例猜；建库以 PRD ER 为唯一权威、以编译/门禁为验收；真岔路（改产物/触红线/PRD 矛盾）停下让人拍板；双角色评审取共识只修真结构，把结论分"改结构/写入纪律/运行时自证"三层；当剩余风险只能运行时验证，就停止空转评审、转去跑起来。**
> 从"页面能跑"到"库能建"，真正的功夫在两处：**把契约在上游收干净，别把返工留给建表；把框架约定用证据钉死，别把黑盒当成想当然。** 把每一步都钉在可复现的记录、字节码与门禁上，而不是"看起来对了就落库"。
