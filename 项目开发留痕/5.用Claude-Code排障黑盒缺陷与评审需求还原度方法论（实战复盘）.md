# 用 Claude Code 排障框架黑盒缺陷、评审需求还原度与自愈开发环境 · 方法论复盘

> 一次真实会话的复盘：**高保真页面交付之后，发现"数据看板 / 吐槽墙全空" → 用系统化排障把根因钉死在框架黑盒里并修复；再用"多角色专家团队"对 PRD 决策基线核对需求还原度与遗漏；中途 `vue-coding` skill 突然加载不了，顺手定位并修复了损坏的插件注册表。**
> 载体是「销售商机互助平台」（Quectel-code 企业框架 + QMonoX + AIRequestGuard 防腐层 + UAA SSO）。本文与《AI 全程搭建 0→1 方法论》《Quectel-code AI 开发正确姿势》《用 Claude Code 做 PRD 评审与产品文档方法论》《用 Claude Code 打造项目 AI 配置三件套方法论》配套——前四份讲"从 0 到 1 把东西做出来"，**这份讲"东西做出来之后，怎么验证它对、纠正它错、以及修好会坏的开发环境本身"**。

---

## 一、结论先行（TL;DR）

**"高保真、看起来对"绝不等于"对"。交付之后真正的风险藏在三处：现象级 bug 的根因埋在框架黑盒里、还原度对错了基准、开发环境本身会悄悄坏。**

- 🐞 **现象不等于根因**：页面全空最容易的反应是"图表组件坏了"。真因却在防腐层库的 mock 分支——它把已是 ViewModel 的数据又跑了一遍 adapter（snake 读 camel），字段全落默认值。**不最小复现锁死，就会去改根本没错的组件。**
- 📐 **还原度要比基线、不是比原型**：原型是"长什么样"，PRD 决策基线（42 FEAT + 决策纪要）才是"该有什么"。静态原型天然覆盖不到**规则/流程型需求**（SLA 计时升级、响应闭环率、超期自动关闭）——恰恰是最容易漏的。
- 🧰 **开发环境会悄悄坏**：`vue-coding` 突然报 `Unknown skill`。第一反应不该是重装，而是查**插件注册表**——真因是一次批量注册脚本的路径拼接 bug 把 `installPath` 写成了坏模板串。
- 🔑 让这三件事可靠落地的，是四条：**现象不猜根因用真实运行时复现 + 还原度对基线做多角色交叉验证 + 环境先诊断状态再动手 + 全程门禁兜底（skill 不在也合规）。**

> 一句话方法论：**现象不猜根因、根因用最小复现钉死在黑盒某一行；还原度不比原型比基线、多角色交叉验证分层遗漏；工具失灵先诊断注册表再修、改前备份改后校验；而这一切的底线是门禁，不是 skill 在不在。**

## 二、正确的主线流程（可照抄）

三条线各一条流水线，可分别照抄：

```
排障线（定位框架黑盒缺陷）
  1. 现象出现 → 先声明"未找根因前不改任何代码"（系统化排障铁律）
  2. 先证伪最可疑对象：图表空≠组件坏——独立渲染测试证明组件对，把嫌疑往数据链路上游赶
  3. 读黑盒库真实实现（.d.ts + 编译产物），别靠文档/直觉猜其 mock/契约行为
  4. 写 20 行"真实运行时最小复现"：import 真库 + 真 mock，跑多个配置场景，用输出钉死真因
  5. 改在"正确的层"，让修复同时是未来的正确形态 → 跑门禁（typecheck/self-check/build）验证

评审线（核对需求还原度与遗漏）
  6. 选对基准：以 PRD 决策基线（FEAT + 决策纪要）为轴，不比原型
  7. 多角色并行 subagent（PM 范围闭环 / 架构 数据契约SSOT / 前端 交互还原），各自独立取证
  8. 交叉验证定可信度：多角色独立命中＝高可信；分层遗漏（建库硬冲突 / 真缺页 / 规则偏差 / 范围待拍板 / 下阶段）
  9. 区分"真缺陷"与"范围内骨架"，冲突点与范围决策交人拍板

环境线（自愈开发环境）
  10. 工具失灵 → 先诊断"状态/注册表"（installed_plugins.json / known_marketplaces.json / .orphaned_at），别乱重装
  11. 备份 → 修坏路径 → node 校验 JSON 合法 → 认清"注册表仅启动时读、必须真重启"
```

## 三、关键方法与技巧复盘（本文精华）

### 技巧 1 · 现象不猜根因——先证伪最可疑对象，把嫌疑往上游赶
"图表全空"最省事的假设是"图表组件坏了"，而这几乎总是错的入口。正确做法是**先证伪**：把 SVG/CSS 图表组件抽出来做一次**独立渲染测试**（喂真数据、截图确认），证明组件完全正常。嫌疑随即被赶到上游——"数据根本没到组件"。系统化排障的铁律是 **NO FIXES WITHOUT ROOT CAUSE**：没锁死根因前，一行业务代码都不改。

### 技巧 2 · 用"真实运行时最小复现"钉死黑盒真因（本文最硬核）
黑盒框架的行为**别靠读文档或直觉猜**。本次真因在防腐层库 `AIRequestGuard` 的 mock 分支：它对已注册的 `viewSchema()`（本项目注册的是 **ViewModel**）**再跑一次 adapter**——adapter 按 `snake_case`（`uv_mom/hot_contents`）去读一个 `camelCase` 对象，字段全落默认值 `0/[]`，于是图表/列表全空。锁死它的方式不是读源码猜，而是写一个 **20 行脚本 import 真实库运行时 + 真 mock**，跑四个配置场景，用输出定案：

| 场景 | 模式 | viewSchema | request | 结果 |
|---|---|---|---|---|
| 当前配置 | mock | ViewModel | — | ❌ 全空（双重转换吞数据）|
| 改 viewSchema | mock | 原始 DTO | — | ❌ 另一层 `pickBySchema` 裁掉 |
| **正解** | **real** | ViewModel | **返回 mock DTO** | ✅ 数据全对 |

一次复现胜过十次猜测——它同时**证明了真因、证伪了另外两种改法、并指出唯一正解**。

### 技巧 3 · 改在"正确的层"——让修复同时是未来的正确形态
定位到真因后，最忌"在库的 mock 模式里绕来绕去打补丁"。正解是把"后端未就位"这件事**下沉到 request 层**：库保持 `real` 模式，用一个 `mockRequest(<DTO切片>, () => 真实请求)` 包裹——mock 时直接返回 DTO，adapter 只转一次即正确；**且这正是后端就绪后的正确形态**（届时一个 `MOCK_ENABLED=false` 开关全量切回真实接口）。好的修复不是"能跑就行"，而是**让当下的修复恰好落在未来正确架构的位置上**。

### 技巧 4 · 还原度别比原型、要比基线——评审的第一原则
交付页面"照原型还原度高"是一回事，"符合需求"是另一回事。**原型只说"长什么样"，PRD 决策基线才定"该有什么"。** 静态原型天然覆盖不到**规则型/流程型需求**——SLA 首响计时与三级升级、需求响应闭环率（北极星）、超期 7 天自动关闭、状态流转完整性——而这些恰是最容易整块漏掉的。评审的轴必须钉在**决策基线（FEAT 台账 + 决策纪要）**，不是原型。

### 技巧 5 · 多角色并行 + 交叉验证定可信度（评审核心）
沿用 PRD 评审那套"多角色专家团队"，评审对象换成"已实现前端 vs 需求基线"：**三个 subagent 各一 lens（PM 范围与闭环 / 架构 数据契约与 SSOT / 前端 交互还原与状态流转），独立并行、各自去源码＋基线取证**。可信度由交叉验证决定：
- **三方独立命中＝高可信**：`PC-09 用户权限页` 与 `PC-26 产品线维护页`整页缺失、需求残留已删的 `deadline` 字段（违背决策 D2）——三个角色不约而同点到，几乎必真。
- **单方但源码坐实＝采纳**：架构独查出公告状态 `expired` 与基线 `archived` 冲突、需求 `status` 大小写在两个模块分裂——建库即撞，采纳。
- **分歧点交主控裁**：PM 说"6 个下期模块进了 v1.0 导航是越界"，前端说"作为展示骨架合规"——真相是**本轮用户有意做的预览页**，是范围决策、不是 bug。

### 技巧 6 · 分层遗漏、区分"真缺陷"与"范围内骨架"
一份能落地的评审结论不是"一堆问题"，而是**按对目标的影响分层**：① 建库/契约硬冲突（服务"建表"目标，最先修）② 真缺页/真遗漏 FEAT ③ PRD 明文修正点未落地 ④ 范围纪律（需人拍板）⑤ 交互硬规则（下阶段）。同时**严格区分**：既定"只做展示层 + mock 骨架"范围下，缺角色门禁/不可逆逻辑**不算 bug**；但缺页、枚举冲突、决策明文点没落地是**真缺陷**。别一刀切算遗漏，也别拿"骨架"豁免真缺页。

### 技巧 7 · 开发环境会悄悄坏——先诊断注册表，别乱重装
`vue-coding` 突然 `Unknown skill`，"重装一遍"是最贵也最可能无效的反应。正确顺序是**先诊断状态文件**：本次一眼看到缓存目录里冒出 `.orphaned_at` 标记，再查 `installed_plugins.json` —— 三个插件的 `installPath` 全被写成坏模板串 `C:Users...cache...${p}${v}`（**模板变量 `${p}/${v}` 未展开 + 反斜杠被吞**），`known_marketplaces.json` 的 `installLocation` 同样坏。真因是一次批量注册脚本的路径拼接 bug，**文件其实都好好在缓存里，纯粹是注册表指错了**。修复三步走：**改前备份两个 JSON → 改回真实路径 + 删 `.orphaned_at` → `node -e require()` 校验 JSON 合法**。并认清一条硬事实：**注册表只在 CLI 进程启动那一刻读入，`/plugin` 不重启进程，必须完全退出再启才生效。**

### 技巧 8 · 门禁是最终真相，skill 只是助手
一个反直觉但关键的观察：这套 `vue-coding` skill **整个会话其实都处于 orphaned、根本没加载**，可所有前端改动（含 19 个文件的 mock 修复）**self-check 全 `0/0`、typecheck/build 全绿**。结论：**合规是靠门禁强制的，不是靠 skill 在不在。** 所以工具链临时失灵时，不必卡死等修好——以门禁（self-check CLI / typecheck / build）为准照常推进，环境择机再修。这也反过来说明：**把约束固化进门禁，比固化进"某个必须在场的工具"更鲁棒。**

## 四、给"交付后验证与纠偏"的协作姿势

| 姿势 | 说明 |
|---|---|
| **现象不猜根因** | 全空先证伪组件、把嫌疑赶到数据链路上游；没锁根因不改代码。 |
| **真实运行时最小复现** | 黑盒行为写 20 行脚本 import 真库＋真数据、跑多场景钉死，别读文档猜。 |
| **改在正确的层** | 让修复恰好落在未来正确架构的位置（mock 下沉 request 层＝后端就绪的切换点）。 |
| **还原度比基线不比原型** | 评审轴钉在 PRD 决策基线；规则/流程型需求是原型盲区，重点查。 |
| **多角色交叉验证** | 三 lens 独立并行取证；多方命中＝高可信、单方源码坐实＝采纳、分歧交人裁。 |
| **分层遗漏、区分真缺陷** | 按对目标影响分层；范围内骨架不算 bug、缺页/契约冲突是真缺陷。 |
| **环境先诊断再动手** | 工具失灵先查注册表/状态文件；改前备份、改后校验、认清"必须真重启"。 |
| **门禁兜底** | 以 self-check/typecheck/build 为最终真相；skill/工具不在也照常合规推进。 |
| **决策留给人** | scope（全量高保真 vs 骨架）、评审深度、下期模块入口——给推荐问用户。 |

## 五、可复用检查清单（开工即用）

**排障阶段**
- [ ] 先声明"未找根因不改代码"；把最可疑对象**证伪**（如独立渲染测试证明组件对）
- [ ] 读黑盒库真实实现（`.d.ts`＋编译产物），别靠文档/直觉猜其行为
- [ ] 写"真实运行时最小复现"，多场景对照，钉死真因＋证伪其他改法
- [ ] 改在正确的层；跑 typecheck/self-check/build 验证到全绿

**评审阶段**
- [ ] 基准选 PRD 决策基线（FEAT＋决策纪要），**不比原型**
- [ ] 每角色一个 subagent（PM/架构/前端…）独立并行、各自取证
- [ ] 交叉验证定可信度；结论**分层**（建库硬冲突 / 缺页 / 规则偏差 / 范围待拍板 / 下阶段）
- [ ] 区分"真缺陷"与"范围内骨架"；分歧与范围决策交人拍板

**环境阶段**
- [ ] 工具失灵先诊断注册表/状态（`installed_plugins.json` / `known_marketplaces.json` / `.orphaned_at`）
- [ ] 改前备份 → 修坏路径 → `node -e require()` 校验 JSON → **完全重启**（非 `/plugin`）

**红线自问**
- [ ] 这是根因还是现象？我最小复现锁死过吗，还是"觉得"是它？
- [ ] 我在比原型还是比基线？规则/流程型需求核过没？
- [ ] 工具没生效，是逻辑错还是"没真重启/注册表坏"？门禁绿了吗？

## 六、效率与成本的诚实备注

- **成本主要花在三处**：① 定位黑盒真因的读源码＋写复现脚本（一次性，但值）；② 3 个评审 subagent（各独立 Opus 上下文、并行取证）；③ 逐文件精确回改（mock 修复 19 个 api 文件）。
- **省钱要点**：
  - **一次复现胜过反复试**：与其"改一版试一版"猜库行为，不如花 20 行脚本把真因＋正解一次钉死——查一次比错三版便宜。
  - **评审并行、主控整合**：3 份评审 background 并行，墙钟＝最慢一份；可信度由主控交叉核对，不让 agent 互聊发散。
  - **门禁兜底省掉"等环境"**：skill 加载不了时不空等，以 self-check 为准继续推进，环境择机修——避免工具故障阻塞交付。
  - **坑升格记忆＝长期资产**：把"mock 双重转换""注册表坏路径必须真重启"写进项目记忆库，后续会话直接命中，省下重新踩坑的开销。
- **可推广性**：这套"证伪→最小复现→改对层"的排障主线、"对基线多角色交叉验证分层"的评审主线、"先诊断注册表再修"的环境自愈主线，适用于**任何"交付后发现现象级 bug / 需要核对还原度与遗漏 / 工具链临时失灵"**的场景，可作团队 SOP。

## 七、一页纸总纲

> **现象不猜根因，用真实运行时最小复现把 bug 钉死在黑盒某一行；还原度不比原型比基线，多角色交叉验证、分层遗漏、区分真缺陷与范围内骨架；工具失灵先诊断注册表再修、改前备份改后校验、必须真重启；而这一切的底线是门禁，不是某个必须在场的 skill。**
> 交付之后的"验证与纠偏"**同样是硬功夫、且价值极高**，前提是：**别拿现象当根因、别拿原型当基线、别拿重装当诊断、别拿 skill 当门禁**——把每一步都钉在可复现的事实与门禁上，而不是"看起来对了就交差"。
